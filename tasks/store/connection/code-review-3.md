## Новые замечания

### CR-12: Паника при повторном SIGTERM/SIGINT [medium]

**Файл:** `cmd/store/main.go`, строки 29–32

```go
go func() {
    <-sig
    close(stopCh)
}()
```

Канал `sig` буферизован на 1. Если пользователь отправляет два сигнала подряд (Ctrl+C дважды — стандартное нетерпеливое поведение в терминале), второй сигнал буферизуется и после завершения первого `<-sig` горутина снова читает из `sig` и вызывает `close(stopCh)` — паника: close of closed channel.

Реальный сценарий: оператор жмёт Ctrl+C, видит что процесс не завершается мгновенно (ждёт активные соединения), жмёт ещё раз.

**Фикс:** `sync.Once` вокруг `close(stopCh)`, либо `select { case stopCh <- struct{}{}: default: }` если изменить stopCh на небуферизованный, либо просто читать в цикле и завершать горутину после первого сигнала:

```go
go func() {
    <-sig
    close(stopCh)
    signal.Stop(sig) // прекращаем принимать сигналы — горутина завершается
}()
```

Самый простой вариант: добавить `signal.Stop(sig)` после `close(stopCh)` — тогда второй сигнал не попадёт в канал и горутина не перечитает.

### CR-13: `isCleanClose` семантически некорректна — маскирует принудительное закрытие как EOF [minor]

**Файл:** `cmd/store/handler.go`, строки 22–24

```go
func isCleanClose(err error) bool {
    return errors.Is(err, syscall.EPIPE) || errors.Is(err, net.ErrClosed)
}
```

`net.ErrClosed` возникает когда мы сами закрыли соединение (через `conn.Close()` из watcher-горутины при `ctx.Done()`). То есть при принудительном shutdown ошибка записи будет `net.ErrClosed` → код логирует "connection closed (EOF)" на уровне DEBUG. Но это не clean close клиента — это наше принудительное закрытие по таймауту.

В текущем MVP это не критично: поведение в обоих случаях одинаковое (return). Но при следующей фиче (реальный роутер, in-flight транзакции) неверное логирование уровня станет источником потерянных событий при диагностике. Семантика функции и её название врут.

**Фикс:** разделить случаи или переименовать функцию в `isSuppressedWriteError` с комментарием что включает как client-side close, так и forced server-side close.
