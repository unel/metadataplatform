# Ретро: store/connection

_Дата: 2026-04-26_  
_Участники: Танк, Ада Лавлейс, Гримм, Кроули, Азирафаль, Харли Куин_

## Что шло хорошо

- Команда нашла реальные баги до выхода в прод: CR-9 (deadlock в graceful shutdown), CR-12 (паника при двойном SIGTERM)
- Recall-артефакты (notes-*/complaints-*) накопили достаточно детали для предметного разбора
- Тест-инфраструктура создана и работает; happy-path и adversarial покрытие есть
- Обсуждение ретро вскрыло системные паттерны, а не разовые инциденты

## Проблемы

### Требования

- **Спека drift от кода с первого же ревью** — источник: Ада, Танк  
  `MaxConnections` добавлен через CR-5 как требование стандарта, в `spec.md` не появился. `bufio.Scanner` описан в спеке (`ErrTooLong`), реализация на `bufio.Reader.ReadLine` (там нет `ErrTooLong`). Следующий читатель спеки получит неверную картину. Нет ни правила, ни момента в процессе где это ловится.

- **Acceptance расширяла спеку вместо верификации** — источник: Танк  
  EDGE-04 (директория для сокета не существует) появилась только на этапе acceptance-write — в спеке этого failure mode не было. Spec-review не задал вопрос "что acceptance может принести нового?". В итоге спека закрылась с дырой, acceptance её закрыла без флага.

- **Full-duplex семантика не описана в acceptance** — источник: Кроули, Азирафаль  
  FT-04 написан в request-response семантике: клиент пишет 3 строки, store отвечает. На 3 строках всё работает — kernel buffer не успевает заполниться. `TestHighThroughput_1000Lines_NoneDropped` повис намертво при первом же запуске: Unix socket full-duplex, клиент не читал ответы параллельно с записью. Нет чек-листа который бы задал вопрос "full-duplex или request-response?" на этапе acceptance-write.

- **Лог-формат не специфицирован** — источник: Танк  
  "В логах появляется запись" — это пожелание, не спека. Реализатор мог написать `fmt.Println`. Конкретный формат лог-строки нигде не зафиксирован.

- **FT-08/FT-09 создают иллюзию coverage** — источник: Азирафаль, Гримм  
  Оба сценария (SIGTERM и SIGINT) идут через `close(stopCh)` в embedded runner — `signal.Notify`-ветка не тестируется в принципе. В acceptance два отдельных сценария, по факту один codepath. Ограничение embedded runner известно, явно нигде не задокументировано.

### Процессные

- **Инварианты в коде не документированы — "минор по стилю" в shutdown-блоке породил регресс** — источник: Ада  
  CR-7: "cancel() дублируется в каждой ветке select — можно вынести". Минорное косметическое замечание. Ада вынесла `cancel()` за select. В CR-9 Гримм нашёл deadlock: в ветке таймаута `cancel()` стал вызываться после блокирующего `<-done` при живых горутинах. Инвариант "cancel() должен выполняться до любого блокирующего ожидания в shutdown ветке таймаута" нигде не был написан. Ни ревьювер, ни реализатор не увидели что косметика меняет семантику.

- **Нет fix-report — история итераций теряется** — источник: Танк  
  14 шагов в status.md, ни одного `spec-fix-report.md` или `code-fix-report.md`. Следующая фича начнёт с теми же слепыми пятнами — документированной памяти о том что было проблемой, не существует.

- **code-fix выдаётся без проверки актуальности** — источник: Ада, Харли  
  code-fix-1 описывал 5 багов. 4 из 5 уже были исправлены до момента выдачи задания. Итерация сгорела впустую.

- **main.go выпал из первых двух раундов ревью** — источник: Гримм  
  Три файла в фиче, `main.go` проверен один раз из трёх. CR-12 (паника при двойном SIGTERM) существовал с первой версии, найден только в третьем раунде. Не был экзотикой — любой нетерпеливый оператор воспроизводит.

### Инструментальные / тестирование

- **`waitForMsg` работал как `time.Sleep` — паттерн нашли в двух местах** — источник: Кроули, Азирафаль  
  `slog` пишет `conn_id` в `attrs`, не в `msg`. `waitForMsg("conn_id")` всегда возвращал `false` и работал как задержка с лишним синтаксисом. После ревью исправили в одном месте, не сделали grep по файлу — тот же паттерн нашли во втором тесте в следующем раунде. Нет стандарта на slog в тестах, нет конвенции "grep по паттерну перед закрытием замечания".

- **`TestGracefulShutdown` зеленел при активном deadlock** — источник: Гримм, Ада, Кроули  
  `TestStore_GracefulShutdown_ForcesCloseAfter5Seconds` использует `sp.Stop()` со своим 6-секундным таймаутом. Тест завершался по внешней обёртке, а не по поведению store. Deadlock внутри — CR-9 — прошёл через test-write и test-review незамеченным. Зелёный тест создал ложную уверенность.

- **testhelpers без собственного review-цикла** — источник: Азирафаль  
  Все 4 итерации test-fix касались вспомогательного слоя, не самих сценариев. testhelpers — фундамент, баг в них делает неверными все тесты поверх. Сейчас они проверяются вместе со сценариями, не как первый приоритет.

- **Acceptance не проверяется на "технически воспроизводима в тесте"** — источник: Кроули  
  NFT-05 требует "IO-ошибку на Unix socket". На Go без `unix.Shutdown` или kill процесса это невоспроизводимо. Сценарий закрыт чистым EOF — формальное покрытие, не реальное. Момент где это должно было остановиться — acceptance-review — не имеет такого чек-пункта.

### Командные

- **Харли как субагент не имеет доступа к `Agent` tool** — источник: Харли, Пользователь  
  Оркестрация требует ручного вмешательства на каждом шаге. Recall занял несколько итераций вместо одной. Флоу не двигается автоматически. Это системное ограничение платформы, не ошибка скилла.

### Находки из внешнего CLAUDE.md (andrej-karpathy-skills)

Три механики которые прямо адресуют наши боли:
1. **Явные критерии верификации на каждом шаге** — `verify: [check]` рядом с каждым шагом задачи. Закрывает тест-симулякры: если в скилле написано "verify: тест падает без реализации", симулякр невозможен.
2. **"State assumptions explicitly"** — при неоднозначности в спеке: формулировать допущение явно, а не выбирать молча. Закрывает drift: добавление `MaxConnections` без флага было бы нарушением.
3. **Формула сильных критериев приёмки** — не "система корректно обрабатывает X", а "тест TestX запускается, получает Y, закрывает соединение — без ошибок". Конкретный assert, не намерение.

## Открытые вопросы

- FT-08/FT-09 через embedded runner: признать ограничение явно в acceptance ("signal handling тестируется только через stopCh, `signal.Notify`-путь не покрыт") или архитектурный рефакторинг через `os.Process.Signal`? Требует решения пользователя.
- Расхождение буфера Unix socket в отчётах (~8 КБ vs ~208 КБ) — установить точное значение из системных источников.
- Пример конфига в спеке содержал поле `ops` следующей фичи — правило "конфиг-пример только текущего MVP" пока нигде не зафиксировано.
