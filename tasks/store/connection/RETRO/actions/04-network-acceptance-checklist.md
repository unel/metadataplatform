# Action: чек-лист для сетевых протоколов в acceptance-write

## Проблема

FT-04 описан в request-response семантике: клиент пишет 3 строки, store отвечает. На 3 строках всё работает. `TestHighThroughput_1000Lines_NoneDropped` повис при первом запуске: Unix socket full-duplex, клиент не читал ответы параллельно с записью, kernel buffer забился, обе стороны зависли. В acceptance не было ни строки про "клиент должен параллельно читать ответы". NFT-05 требует "IO-ошибку на Unix socket" — на Go без `unix.Shutdown` это невоспроизводимо, закрыто чистым EOF как отписка.

## Решение

Добавить в скилл `acceptance-write` обязательный чек-лист для любой фичи с сетевым взаимодействием:

```
Чек-лист для сетевых протоколов:
1. Full-duplex или half-duplex? Кто инициирует поток данных?
2. Что происходит если одна сторона перестаёт читать?
3. Есть ли backpressure? Как проявляется при заполнении буфера?
4. Что при объёме данных > kernel socket buffer (~8 КБ для Unix socket)?
5. Кто закрывает соединение и в каком порядке?
6. Как воспроизвести каждый сценарий в коде теста — инструменты, mock-точки, границы?
   Если воспроизведение невозможно → пометить "manual only" или "требует mock на уровне X"
```

Пункт 6 обязателен: acceptance не должна писать сценарии которые нельзя протестировать автоматически без явной пометки.
