# Acceptance — store/connection

## Сценарии

---

### FT-01 — Старт с валидным конфигом, один сокет

**Given** конфиг содержит `store.sockets` с одним записью (`path: /tmp/test-store.sock`)  
**When** store запускается  
**Then** файл `/tmp/test-store.sock` существует на ФС  
**And** store принимает входящие unix-подключения на этом пути

---

### FT-02 — Старт с валидным конфигом, несколько сокетов

**Given** конфиг содержит `store.sockets` с тремя записями (разные пути)  
**When** store запускается  
**Then** каждый из трёх сокет-файлов существует на ФС  
**And** store принимает подключения на каждом из них независимо

---

### FT-03 — Эхо-ответ на сообщение

**Given** store запущен с одним сокетом  
**And** клиент подключён к сокету  
**When** клиент отправляет строку `{"op":"ping"}\n`  
**Then** store возвращает ровно ту же строку `{"op":"ping"}\n`

---

### FT-04 — Несколько сообщений в одном соединении

**Given** store запущен, клиент подключён  
**When** клиент последовательно отправляет три строки  
**Then** store возвращает каждую строку эхом в том же порядке

---

### FT-05 — Логирование каждой полученной строки

**Given** store запущен  
**And** клиент подключён  
**When** клиент отправляет строку  
**Then** в stdout/stderr store появляется запись уровня DEBUG с содержимым этой строки  
**And** запись содержит conn_id соединения, от которого пришла строка

---

### FT-06 — Логирование старта сокета

**Given** конфиг с двумя сокетами  
**When** store запускается  
**Then** в логах появляется путь каждого поднятого сокета

---

### FT-07 — Логирование нового подключения

**Given** store запущен  
**When** клиент устанавливает соединение  
**Then** в логах появляется запись о новом подключении, содержащая путь сокета, на котором установлено соединение  
**And** запись содержит conn_id этого соединения

---

### FT-08 — Graceful shutdown по SIGTERM

**Given** store запущен, сокет-файл существует  
**When** процессу отправляется SIGTERM  
**Then** в логах появляется запись о завершении работы  
**And** сокет-файл удалён с ФС  
**And** процесс завершается с кодом 0

---

### FT-09 — Graceful shutdown по SIGINT

**Given** store запущен, сокет-файл существует  
**When** процессу отправляется SIGINT  
**Then** в логах появляется запись о завершении работы  
**And** сокет-файл удалён с ФС  
**And** процесс завершается с кодом 0

---

### FT-10 — Fail-fast: конфиг не найден

**Given** путь к конфигу не существует  
**When** store запускается  
**Then** процесс завершается с exit code 1  
**And** в логах есть сообщение об ошибке (fatal)

---

### FT-11 — Fail-fast: конфиг невалидный YAML

**Given** файл конфига существует но содержит невалидный YAML  
**When** store запускается  
**Then** процесс завершается с exit code 1  
**And** в логах есть сообщение об ошибке

---

### FT-12 — Fail-fast: список sockets пустой

**Given** конфиг валидный, но `store.sockets: []`  
**When** store запускается  
**Then** процесс завершается с exit code 1  
**And** в логах есть сообщение об ошибке

---

### FT-13 — Уникальность conn_id при конкурентных подключениях

**Given** store запущен  
**When** десять клиентов подключаются одновременно  
**Then** в логах для каждого из десяти подключений присутствует conn_id  
**And** все десять conn_id различны между собой

---

### FT-14 — Configurable max_line_bytes

**Given** store запущен с конфигом, содержащим `max_line_bytes: 1024`  
**And** клиент подключён  
**When** клиент отправляет поток данных длиной 1025 байт без символа `\n`  
**Then** store закрывает соединение с этим клиентом  
**And** в логах появляется запись об ошибке превышения лимита  
**And** остальные подключённые клиенты продолжают работу без прерывания

---

### NFT-01 — Клиент отключается в середине потока

**Given** store запущен  
**And** два клиента подключены  
**When** первый клиент отправляет строку и принудительно закрывает соединение не ожидая ответа  
**Then** store логирует событие обрыва  
**And** второй клиент продолжает получать эхо-ответы без прерывания

---

### NFT-02 — Сокет-файл уже существует при старте (stale socket)

**Given** на ФС уже есть файл по пути сокета (остался от предыдущего крэша)  
**When** store запускается  
**Then** старый файл удаляется и пересоздаётся  
**And** store успешно принимает подключения  
**And** процесс не завершается с ошибкой

---

### NFT-03 — Параллельная обработка нескольких клиентов

**Given** store запущен  
**When** десять клиентов подключаются одновременно и каждый отправляет сообщение  
**Then** каждый клиент получает эхо своего сообщения  
**And** медленный клиент (задержка перед чтением ответа) не блокирует остальных

---

### NFT-04 — Логирование ошибок чтения/записи

**Given** store запущен, клиент подключён  
**When** клиент принудительно закрывает соединение в момент когда store пишет ответ  
**Then** в логах store появляется запись об ошибке записи  
**And** store продолжает работу (не крашится)

---

### NFT-05 — Уровни лога при закрытии соединений

**Given** store запущен, клиент подключён  
**When** клиент закрывает соединение штатно (EOF)  
**Then** в логах появляется запись уровня DEBUG о закрытии соединения

**Given** store запущен, клиент подключён  
**When** соединение обрывается с IO-ошибкой (неожиданное закрытие)  
**Then** в логах появляется запись уровня INFO о закрытии соединения

---

### NFT-06 — Graceful shutdown ждёт активные соединения

**Given** store запущен  
**And** клиент установил соединение и начал обработку (соединение открыто)  
**When** процессу отправляется SIGTERM  
**Then** store не закрывает соединение немедленно  
**And** store ждёт завершения активного соединения  
**And** после закрытия соединения процесс завершается с кодом 0  
**And** если соединение не закрылось в течение 5 секунд — процесс всё равно завершается с кодом 0, принудительно разорвав его

---

### EDGE-01 — Пустая строка от клиента

**Given** store запущен, клиент подключён  
**When** клиент отправляет пустую строку `\n`  
**Then** store не крашится  
**And** пустая строка пропускается: ответ клиенту не отправляется, в логах запись не создаётся

---

### EDGE-01b — Строка из пробельных символов

**Given** store запущен, клиент подключён  
**When** клиент отправляет строку, состоящую только из пробелов и/или табуляций (`   \n`, `\t\t\n`)  
**Then** store не крашится  
**And** строка пропускается: ответ клиенту не отправляется, в логах запись не создаётся

---

### EDGE-02 — Строка превышает лимит буфера (1 МБ)

**Given** store запущен, клиент подключён  
**When** клиент отправляет поток данных длиной более 1 048 576 байт без символа `\n`  
**Then** store закрывает соединение с этим клиентом  
**And** клиент получает EOF — никаких данных перед закрытием не передаётся  
**And** в логах store появляется запись об ошибке (превышен лимит буфера)  
**And** остальные подключённые клиенты продолжают работу без прерывания

---

### EDGE-03 — Несколько подключений к разным сокетам одновременно

**Given** store запущен с двумя сокетами (`sock-a`, `sock-b`)  
**When** клиент A подключается к `sock-a`, клиент B к `sock-b`, оба отправляют сообщения  
**Then** каждый получает эхо своего сообщения через свой сокет  
**And** трафик между сокетами не смешивается

---

### EDGE-04 — Директория для сокета не существует

**Given** конфиг указывает путь `/run/platform/nonexistent-dir/store.sock`  
**And** директория `/run/platform/nonexistent-dir/` не существует  
**When** store запускается  
**Then** процесс завершается с exit code 1  
**And** в логах есть понятное сообщение об ошибке создания сокета
