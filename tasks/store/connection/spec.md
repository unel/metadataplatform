# Spec — store/connection

## Функциональные требования

### Happy path
- Store читает конфиг и поднимает Unix socket(ы) согласно списку `store.sockets`
- Принимает входящие подключения на каждом сокете
- Читает входящие сообщения построчно (JSONL), логирует каждую строку
- Отвечает эхом: возвращает полученную строку обратно клиенту
- При завершении (SIGTERM / SIGINT) сокет-файлы удаляются

### Fail-fast при старте
- Конфиг не найден или невалидный — `fatal + exit(1)`
- Список `store.sockets` пустой — `fatal + exit(1)`

### Ограничения (не входит в MVP этой фичи)
- Разбор и исполнение операций (`op: query/upsert/delete`) — следующая фича
- Фильтрация по разрешённым `ops` на сокет — следующая фича
- DB-соединение — отдельная фича

## Нефункциональные требования

### Надёжность
- Если клиент отключается в середине чтения — сервер логирует событие и продолжает принимать других клиентов
- Если сокет-файл уже существует при старте (crash без cleanup) — файл удаляется и пересоздаётся, не паникуем

### Параллелизм
- Каждое входящее подключение обрабатывается в отдельной горутине — медленный или зависший клиент не блокирует остальных

### Наблюдаемость
- Логируется: путь каждого поднятого сокета, каждое новое подключение, каждая полученная строка, ошибки чтения/записи, завершение работы

### Сопровождаемость
- Обработчик сообщения (сейчас эхо) — отдельная функция с понятной сигнатурой, чтобы следующая фича просто подменила её на реальный роутер

## Взаимодействие с другими компонентами
- Конфиг: `store.sockets[].path` — пути к сокет-файлам
- Остальные компоненты (api, spawner) подключаются к этим сокетам — но это их задача
