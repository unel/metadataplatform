# Spec — store/connection

## Функциональные требования

### Happy path
- Store читает конфиг и поднимает Unix socket(ы) согласно списку `store.sockets`
- Принимает входящие подключения на каждом сокете
- Читает входящие сообщения построчно (JSONL); каждая строка логируется на уровне DEBUG
- Отвечает эхом: возвращает полученную строку обратно клиенту
- При завершении (SIGTERM / SIGINT) выполняется graceful shutdown (см. ниже), сокет-файлы удаляются

### Fail-fast при старте
- Конфиг не найден или невалидный — `fatal + exit(1)`
- Список `store.sockets` пустой — `fatal + exit(1)`

### Поведение при чтении: буферизованный построчный reader

- Store читает каждое соединение через буферизованный построчный reader (`bufio.Scanner` или эквивалент)
- Максимальный размер одной строки (до символа `\n`) задаётся параметром конфига `store.max_line_bytes` (дефолт: **1 048 576 байт**, 1 МБ)
- Если клиент отправляет данные, превышающие `max_line_bytes` без символа `\n`:
  - соединение принудительно закрывается со стороны store
  - клиент получает EOF, ответное сообщение об ошибке не отправляется
  - в лог пишется ошибка уровня INFO с идентификатором соединения и сообщением о переполнении буфера; точный размер полученных данных не логируется (недоступен при `ErrTooLong`)
  - другие соединения не затрагиваются
- Лимит применяется на каждую строку независимо; клиент может отправлять любое количество строк в рамках одного соединения
- Пустые строки (только `\n`) и строки, состоящие только из пробельных символов, **пропускаются**: ответ клиенту не отправляется, в лог запись не создаётся. Обоснование: пустые строки допустимы как heartbeat-паттерн, трассировка их на уровне INFO создаёт шум без пользы

### Graceful shutdown

- По получении SIGTERM / SIGINT store прекращает принимать новые подключения на всех сокетах
- Сокет-файл удаляется сразу после остановки listener'а (до начала ожидания активных горутин)
- Активным горутинам-обработчикам даётся **5 секунд** на завершение через `context`-based cancellation
- Если по истечении 5 секунд остались активные соединения — они принудительно закрываются, store завершает работу

Порядок операций при shutdown:
1. listener останавливается (перестаёт принимать новые соединения)
2. сокет-файл удаляется сразу после остановки listener'а
3. ждём завершения активных горутин 5 секунд

### Идентификатор соединения

- Каждому принятому соединению присваивается `conn_id` — глобальный монотонно возрастающий счётчик (uint64), сбрасывается при рестарте процесса
- Счётчик `conn_id` инкрементируется атомарно (`sync/atomic`)
- `conn_id` фигурирует во всех лог-записях, связанных с данным соединением
- Формат обеспечивает читаемость в логах; уникальность между рестартами не требуется

### Ограничения (не входит в MVP этой фичи)
- Разбор и исполнение операций (`op: query/upsert/delete`) — следующая фича
- Фильтрация по разрешённым `ops` на сокет — следующая фича
- DB-соединение — отдельная фича
- Защита от одновременного запуска нескольких инстансов store (advisory lock) — отдельная фича
- Валидация входящего JSON — следующая фича; текущий обработчик возвращает строку as-is вне зависимости от содержимого

## Нефункциональные требования

### Надёжность
- Если клиент отключается в середине чтения — сервер логирует событие и продолжает принимать других клиентов
  - нормальное закрытие (клиент прислал EOF) — уровень DEBUG
  - неожиданное отключение (ошибка IO) — уровень INFO
- Если сокет-файл уже существует при старте (crash без cleanup) — файл удаляется и пересоздаётся, не паникуем

### Параллелизм
- Каждое входящее подключение обрабатывается в отдельной горутине — медленный или зависший клиент не блокирует остальных

### Наблюдаемость
- INFO: путь каждого поднятого сокета, каждое новое подключение (с `conn_id`), ошибки чтения/записи, неожиданные отключения (IO error), переполнение буфера, завершение работы
- DEBUG: каждая полученная строка (с `conn_id`), нормальное закрытие соединения (клиент прислал EOF)

### Сопровождаемость
- Обработчик сообщения (сейчас эхо) — отдельная функция с понятной сигнатурой, чтобы следующая фича просто подменила её на реальный роутер

## Конфиг: пример

```yaml
store:
  db_url: postgres://localhost/platform  # перебивается $DB_URL
  max_line_bytes: 1048576                # 1 МБ, перебивается $MAX_LINE_BYTES
  sockets:
    - path: /run/platform/store.sock
      ops: [query, upsert, delete]
    - path: /run/platform/store-readonly.sock
      ops: [query]
    - path: /run/platform/store-write.sock
      ops: [upsert, delete]
```

## Взаимодействие с другими компонентами
- Конфиг: `store.sockets[].path` — пути к сокет-файлам
- Остальные компоненты (api, spawner) подключаются к этим сокетам — но это их задача
