# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Делегирование скиллов агентам

Когда ты получаешь текст скилла/команды (через Skill tool или как развёрнутый prompt), **первым делом проверь первую строку**:

```
_Исполнитель: агент **<Имя>** (`<subagent_type>`)._
```

Если такая строка есть — **ОБЯЗАТЕЛЬНО** запусти `Agent` tool с `subagent_type: <subagent_type>`, передав в `prompt` весь текст команды (включая строку с исполнителем — агент сам её игнорирует). Не выполняй команду инлайн самостоятельно.

Пример: строка `_Исполнитель: агент **Харли** (`harley`)._` → запусти `Agent(subagent_type="harley", prompt=<полный текст команды>)`.

**Исключение для агента `ada`:** всегда спауни с `mode: "bypassPermissions"` — Ада пишет файлы без лишних промптов.

Исключение: если агент уже запущен в текущей сессии и ты им являешься — выполняй инструкции напрямую.

## Выполнение шагов workflow через скиллы

Перед тем как выполнять шаг workflow (`test-write`, `spec-review`, `acceptance-write` и т.д.) — проверь, есть ли одноимённый скилл. Если есть — вызывай его через `Skill` tool, не реализуй логику самостоятельно.

## Продолжение агента после подтверждения пользователя

Субагенты не могут ждать пользователя — они завершаются после своего ответа. Если агент показал план или черновик и спросил подтверждения, а пользователь ответил утвердительно ("да", "погнали", "давай") или дал правки — **используй `SendMessage` к тому же агенту** (по `agentId` из его последнего ответа), не спавни нового. Новый агент не имеет контекста и начнёт с нуля.

Это правило распространяется на **любой ответ пользователя** на вопрос агента, включая уточнения, правки, отказы. Всё что относится к текущему диалогу с агентом — идёт через `SendMessage`, а не через новый `Agent`.

## Workflow: статусы задач и каскадный сброс

Шаги в `tasks/*/status.md` образуют цепочку: каждый артефакт проходит цикл **write → review**. Если артефакт переписывается после того как ревью уже было пройдено — необходимо сбросить статусы по каскаду:

- Ревью переписанного артефакта: `done` → `pending`
- Все последующие шаги цепочки: `done` → `needs-recheck`

Это применяется к любому артефакту: спеке, критериям приёмки, тестам, коду. Не только к спеке.

Значения статусов:
- `done` — шаг завершён и актуален
- `pending` — шаг нужно выполнить
- `needs-recheck` — шаг был выполнен, но вышестоящий артефакт изменился; нужно перепроверить перед продолжением
- `failed` — шаг выполнен, но не прошёл ревью

При чтении `status.md` в `return-to-feature`: шаги со статусом `needs-recheck` трактуются как **не выполненные** — их нужно повторить в порядке цепочки, начиная с первого `needs-recheck`.
