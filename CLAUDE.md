## Делегирование скиллов агентам

Когда ты получаешь текст скилла/команды (через Skill tool или как развёрнутый prompt), **первым делом проверь первую строку**.

### Формат `_Исполнитель: агент_` — делегировать субагенту

```
_Исполнитель: агент **<Имя>** (`<subagent_type>`)._
```

Запусти `Agent` tool с `subagent_type: <subagent_type>`, передав в `prompt` весь текст команды. Не выполняй инлайн самостоятельно.

**Исключение для агента `ada`:** всегда спауни с `mode: "bypassPermissions"` — Ада пишет файлы без лишних промптов.

Исключение: если агент уже запущен в текущей сессии и ты им являешься — выполняй инструкции напрямую.

### Формат `_Режим: **Харли Куин**_` — выполнять инлайн с имперсонацией

```
_Режим: **Харли Куин**. Выполняется оператором напрямую._
```

Выполняй скилл **сам**, имперсонируя Харли Куин: её стиль, тон, подход к оркестрации. Субагент не нужен — ты main thread и можешь спаунить агентов через `Agent` tool напрямую. Читай `AGENT.md` Харли для калибровки стиля если нужно.

## Выполнение шагов workflow через скиллы

Перед тем как выполнять шаг workflow (`test-write`, `spec-review`, `acceptance-write` и т.д.) — проверь, есть ли одноимённый скилл. Если есть — вызывай его через `Skill` tool, не реализуй логику самостоятельно.

## Продолжение агента после подтверждения пользователя

Субагенты не могут ждать пользователя — они завершаются после своего ответа. Если агент показал план или черновик и спросил подтверждения, а пользователь ответил утвердительно ("да", "погнали", "давай") или дал правки — **используй `SendMessage` к тому же агенту** (по `agentId` из его последнего ответа), не спавни нового. Новый агент не имеет контекста и начнёт с нуля.

Это правило распространяется на **любой ответ пользователя** на вопрос агента, включая уточнения, правки, отказы. Всё что относится к текущему диалогу с агентом — идёт через `SendMessage`, а не через новый `Agent`.

## Workflow v2: статусы и каскадный сброс

В v2 каждый процесс ведёт свой лог: `tasks/$FEATURE/processes/$PROCESS/status-log.md`.

Формат записи:
```
# {{datetime}} — {{статус}}
{{опциональный комментарий}}
```

**Текущий статус процесса** — последняя запись в его `status-log.md`.

### Значения статусов

| Статус | Смысл |
|---|---|
| `pending` | ожидает запуска |
| `in-progress` | выполняется (или прервалось — требует внимания) |
| `done` | завершён и актуален |
| `failed` | завершён, но не прошёл ревью → нужен fix |
| `stale` | был `done`, но upstream-артефакт изменился → повторить |
| `clarification` | процесс остановился: upstream-артефакт неполон или противоречив |

### Каскадный сброс

После fix — добавь запись `stale` во все последующие процессы которые были `done`:

```
# {{datetime}} — stale
Upstream артефакт изменён: <что именно>. Требует повторного прохода.
```

Цепочка зависимостей (каждый шаг зависит от всего выше):
```
01-spec/01-write
  → 01-spec/02-review
  → 02-acceptance/01-write → 02-acceptance/02-review
  → 03-tests/01-write → 03-tests/02-review → 03-tests/04-run
  → 04-code/01-write → 04-code/02-review → 04-code/04-testing
  → 05-docs/01-write → 05-docs/02-review
```

`stale` трактуется как «не выполнен» — повторить начиная с первого `stale` в порядке pipeline.

### Clarification

Процесс в `clarification` — сигнал что upstream-артефакт неполон или противоречив. Алгоритм отката:

1. Прочитай report процесса — там написано что неясно
2. Определи откат: acceptance → `02-acceptance/03-fix`; spec → `01-spec/03-fix`; нужно у пользователя — стоп, задай вопрос
3. Добавь `in-progress` в `status-log.md` целевого upstream-шага
4. Добавь `stale` последующим шагам той же группы (если были `done`/`failed`)
5. Добавь `stale` всем группам начиная с группы в `clarification`
6. Запусти upstream fix-скилл
