# store — технические детали

## Архитектура

```
main()
  └─ RunStoreWithExitCode()     — загрузка конфига, валидация, запуск
       └─ RunStore()            — жизненный цикл: listener'ы, semaphore, shutdown
            └─ acceptLoop()     — горутина на каждый сокет, принимает соединения
                 └─ handleConn() — горутина на каждое соединение, читает строки
                      └─ handleMessage() — обработчик одной строки (сейчас: эхо)
```

### Компоненты

**`main.go`** — точка входа. Разбирает флаги (`--config`), настраивает перехват сигналов (`SIGTERM`, `SIGINT`) через `stopCh chan struct{}`, передаёт управление в `RunStoreWithExitCode`. Реализует `slogAdapter` — адаптер интерфейса `logger` к `log/slog`.

**`config.go`** — структуры конфига, загрузка из YAML, применение дефолтов. Переменные окружения (`$CONFIG_PATH`) обрабатываются в `main`, не в `loadConfig`.

**`store.go`** — ядро сервиса:
- `RunStoreWithExitCode` — обёртка с загрузкой конфига и возвратом exit-кода
- `RunStore` — создаёт listener'ы, семафор соединений, запускает `acceptLoop` в горутинах, ждёт `stopCh`, выполняет shutdown
- `acceptLoop` — цикл `Accept()` для одного сокета; контролирует семафор; назначает `conn_id`
- `listenUnix` — создаёт Unix-сокет; если файл уже существует — удаляет и пересоздаёт

**`handler.go`** — логика обработки соединения:
- `handleConn` — буферизованное построчное чтение, обработка ошибок, запись ответа
- `handleMessage` — единственная точка подмены бизнес-логики (сейчас возвращает строку as-is)

**Тесты** — `connection_test.go`, `connection_adversarial_test.go`, `testhelpers_test.go`. Используют `logger` interface (`logBuffer`) и `RunStore` напрямую, без системных сокетов.

## Протокол

### Формат

- Транспорт: Unix domain socket (SOCK_STREAM)
- Кодировка: JSONL — одно сообщение на строку, завершается `\n`
- Текущая версия не валидирует JSON — принимается любая строка

### Запрос / ответ

```
client -> server:  {"op":"query","table":"assets"}\n
server -> client:  {"op":"query","table":"assets"}\n
```

Одна строка входа → одна строка ответа. Множество строк в рамках одного соединения — допустимо.

### Пустые строки

Строки, состоящие только из пробельных символов (`\n`, `\r\n`, пробелы), игнорируются: ответ не отправляется, в лог не пишется. Это поддерживает heartbeat-паттерн.

### Переполнение буфера

Если строка превышает `max_line_bytes` (дефолт: 1 048 576 байт) без символа `\n`:
- соединение закрывается немедленно
- клиент получает EOF без сообщения об ошибке
- в лог пишется INFO с `conn_id`

Детектирование происходит во время накопления чанков через `bufio.Reader.ReadLine()` — как только накопленный размер превышает лимит, соединение закрывается без дочитывания.

### Ошибки записи

| Ошибка | Интерпретация | Действие |
|--------|---------------|----------|
| `EPIPE` | клиент закрыл соединение пока мы отвечали | DEBUG-лог, закрытие |
| `net.ErrClosed` | принудительное закрытие при shutdown | INFO-лог, закрытие |
| прочие | IO-ошибка | INFO-лог, закрытие |

## Graceful shutdown

Порядок операций при получении `SIGTERM` / `SIGINT`:

1. `stopCh` закрывается в `main`
2. `RunStore` получает сигнал через `<-stopCh`
3. Для каждого listener'а: `ln.Close()` → сокет-файл удаляется (`os.Remove`)
4. `acceptLoop` завершается — `ln.Accept()` возвращает ошибку
5. Ждём завершения активных `handleConn`-горутин через `wg.Wait()`, таймаут 5 секунд
6. При таймауте: `context.cancel()` → watcher-горутина в каждой `handleConn` вызывает `conn.Close()` → горутины завершаются → ждём `<-done`

Context cancellation в `handleConn`: при старте каждой горутины запускается watcher, который ждёт `ctx.Done()` или закрытия `connClosed`. Это позволяет прервать блокирующий `Read` через закрытие соединения.

## Ограничения

| Параметр | Значение | Источник |
|----------|----------|---------|
| Макс. размер строки | 1 048 576 байт (1 МБ) | `max_line_bytes` в конфиге |
| Макс. соединений | 1024 | `max_connections` в конфиге |
| Таймаут shutdown | 5 секунд | захардкожено в `RunStore` |
| Протокол | только Unix domain socket | архитектурное решение |

### Что не поддерживается в текущей версии

- Разбор и маршрутизация операций (`op: query/upsert/delete`) — следующая фича
- Фильтрация разрешённых `ops` на уровне сокета — следующая фича
- Подключение к базе данных — отдельная фича
- Валидация входящего JSON — следующая фича
- TCP/TLS-транспорт — не предусмотрен
- Защита от одновременного запуска нескольких инстансов (advisory lock) — отдельная фича

При превышении `max_connections` новое соединение отклоняется немедленно (WARN в лог), без очереди ожидания.

## Точки расширения

**Подмена бизнес-логики** — `handleMessage(line string) string` в `handler.go`. Сигнатура намеренно простая: принимает строку, возвращает строку. Следующая фича заменит эхо на реальный роутер запросов.

**Добавление нового сокета** — только конфиг (`store.sockets`), код не меняется.

**Логирование** — `logger` interface (`add(level, msg string, attrs map[string]string)`). Любая реализация подходит: `slogAdapter` в проде, `logBuffer` в тестах.

**Конфигурация лимитов** — `max_line_bytes` и `max_connections` в YAML. Таймаут shutdown в 5 секунд потребует изменения в `RunStore` если понадобится сделать его настраиваемым.
