_Исполнитель: агент **Танк** (`Танк`)._

Проведи архитектурное обсуждение и зафикисруй решение в RFC документе.

Аргументы: `$ARGUMENTS` — путь к фиче или тема, плюс контекст из status.md если есть

## Шаги

1. Прочитай `tasks/$ARGUMENTS/status.md` — найди `needs-rfc` с причиной и контекстом
2. Прочитай `CLAUDE.md`, `tasks/$ARGUMENTS/spec.md`, `deps.md` — собери полный контекст
3. Прочитай уже принятые ADR в `docs/tech/adr/` — нет ли уже решения этого вопроса
4. Прочитай шаблон `docs/skills-guide/templates/rfc.md`, заполни по контексту

5. Покажи RFC пользователю, жди ответа

6. Цикл обсуждения — повторяй пока не принято решение:
   - пользователь задаёт вопрос или предлагает новый вариант → обнови RFC (добавь вариант или уточни существующий), добавь тезис в секцию `## Обсуждение`, покажи обновлённый RFC
   - пользователь выбрал вариант или описал решение → переходи к шагу 7

```markdown
## Обсуждение
- Пользователь: "а что если добавить кэш?"
  → Добавлен Вариант 3: кэширование на уровне store
- Пользователь: "вариант 3 усложняет store"
  → Минусы варианта 3 уточнены
```

7. После принятия решения:
   - добавь в RFC секцию `## Решение` с описанием и обоснованием
   - обнови `Статус: принято`
   - сохрани в `docs/tech/rfc/<kebab-name>.md`
   - верни вердикт: `rfc-done: <название RFC>, decision: <краткое решение>`

## Важно

Не запускать `/adr` самостоятельно — вернуть вердикт и дать вызывающему решить.
