Проверь спеку на противоречия, сложности и corner-cases.

Аргументы: `$ARGUMENTS` — путь к фиче, например `store/query`

## Шаги

1. Прочитай `CLAUDE.md` и `tasks/$ARGUMENTS/spec.md`
2. Если spec.md не содержит и ФТ и НФТ — сообщи что сначала нужны `/spec-ft` и `/spec-nft`
3. Запусти `/deps-spec $ARGUMENTS` — получи список смежных фич, прочитай их `spec.md`
4. Проверь по чеклисту:

**Противоречия:**
- требования не противоречат друг другу внутри спеки
- нет противоречий со спеками смежных фич (`conflicts` в deps)
- нет противоречий с архитектурой в `CLAUDE.md`

**Полнота ФТ:**
- все входные данные описаны
- описано поведение при невалидных входных данных
- описано поведение при недоступности зависимостей
- явно обозначено что не входит в MVP

**Полнота НФТ:**
- покрыты все релевантные категории из `/spec-nft`
- НФТ конкретны и измеримы (не "должно быть быстро")

**Corner-cases:**
- пустые входные данные
- граничные значения
- конкурентные запросы если применимо
- частичный сбой в середине операции

5. Определи N: посчитай существующие файлы `tasks/$ARGUMENTS/spec-review-*.md`, N = количество + 1
6. Запиши результат ревью в `tasks/$ARGUMENTS/spec-review-N.md`:
   - если проблем нет — "Спека чистая. Готово к acceptance."
   - если есть проблемы — список с классификацией и объяснением для каждой
7. Обнови `tasks/$ARGUMENTS/status.md`:
   - `spec-review: done` если проблем нет
   - `spec-review: failed` если есть проблемы (без деталей — они в spec-review.md)
8. Выведи итог пользователю

## Важно

Не переписывать спеку самостоятельно — только указывать проблемы и ждать решения.
Если обнаружен архитектурный вопрос — предложить `/rfc`.
