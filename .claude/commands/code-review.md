Проверь код на соответствие спеке, НФТ и стандартам.

Аргументы: `$ARGUMENTS` — путь к фиче, например `store/query`

## Шаги

1. Прочитай `tasks/$ARGUMENTS/spec.md`, код фичи и `tasks/$ARGUMENTS/status.md`
2. Если `code-write: done` нет в `status.md` — сообщи что сначала нужен `/code-write` и остановись
3. Прочитай `docs/standards/` — список папок, для каждой прочитай `description.md`

4. **Составь чеклист** конкретно для этих изменений:
   - Выдели только те пункты из общих стандартов, которые релевантны данному коду
   - Добавь специфичные пункты исходя из типа изменений (новый query — проверить N+1 и индексы; новый endpoint — проверить валидацию; UI компонент — проверить типы и design tokens и т.д.)

5. **Пройдись по чеклисту** — для каждого пункта зафиксируй статус и найденные проблемы

6. Классифицируй найденные проблемы:
   - **критичные** — нарушают ФТ или НФТ, обязательно фиксить
   - **важные** — нарушают стандарты, желательно фиксить
   - **минорные** — стиль, читаемость, на усмотрение

7. Если возник архитектурный вопрос — вызови `/ask`, обработай вердикт согласно `docs/skills-guide/uncertainty-handling.md`

8. Запиши в `tasks/$ARGUMENTS/status.md`:
   - `code-review: done` если нет критичных проблем
   - `code-review: failed` + список проблем с классификацией если есть критичные

## Важно

Не исправлять код самостоятельно — только описывать проблемы.
Решение о том куда двигаться принимает оркестратор.
