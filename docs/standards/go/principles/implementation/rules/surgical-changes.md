# Правило: surgical changes

## Суть

Каждое изменение трассируется к конкретному пункту `acceptance.md`. Изменения без трассировки не добавляются.

## Формат трассировки

При написании кода держи в голове (или явно в комментарии к PR):

```
изменение → acceptance criterion → сценарий
```

Пример:
```
conn.go: обработка ErrRefused → AC-3 → "connect_refused: возвращает ErrConnectionRefused"
timeout.go: константа DefaultTimeout → AC-5 → "connect_timeout: завершается за ≤200мс"
```

## Почему "хирургически"

Хирург не правит то, к чему не прикасается скальпель. Изменение кода, не связанного с задачей:

1. **Скрывает намерение** — ревьювер не понимает зачем это изменение
2. **Ломает тесты** — код был стабильным по неочевидной причине
3. **Усложняет ревертинг** — нельзя откатить только нужную часть
4. **Создаёт шум в diff** — важное прячется среди косметики

## Плохо

```go
// задача: добавить обработку timeout
// попутно переименовал переменную i → idx, вынес константу, добавил комментарий
func connect(addr string) error {
    for idx := 0; idx < MaxRetries; idx++ { // раньше было i
        // пробуем подключиться
        if err := dial(addr); err != nil {
            ...
        }
    }
}
```

## Хорошо

```go
// задача: добавить обработку timeout — только это
func connect(addr string, timeout time.Duration) error {
    ctx, cancel := context.WithTimeout(context.Background(), timeout)
    defer cancel()
    return dial(ctx, addr)
}
```

Соседний код с i/idx — не трогаем. Если хочется улучшить — отдельный PR, отдельный AC.

## Исключение

Фикс, который технически невозможно сделать не затронув соседний код (например, разделение большого модуля), — фиксируется явно в отчёте: "затронул X потому что Y требовало разбиения модуля для AC-N".
