# Принципы реализации в Go

Эти правила описывают дисциплину написания кода в контексте задачи. Цель — чтобы каждое изменение было обоснованным, трассируемым и ограниченным тем, что явно запрошено.

## Четыре принципа

### 1. Surgical changes (хирургические изменения)

Каждое изменение в коде трассируется к конкретному пункту `acceptance.md`. Если изменение нельзя сопоставить с acceptance — оно не добавляется.

Нет: "заодно улучшил именование в соседней функции"  
Нет: "добавил обработку случая X, он логично следует из Y"  
Да: "обработка ErrTimeout — AC-7, сценарий `connect_timeout`"

Подробнее — в `rules/surgical-changes.md`.

### 2. No adjacent code (не трогай соседний код)

Код, который не связан с текущей задачей, не изменяется — даже если он выглядит плохо. Рефакторинг соседнего кода:
- смешивает назначение коммита
- усложняет ревью
- может сломать тесты которые ты не видел

Если нашёл проблему в соседнем коде — зафикси в issue или заметке, но не правь в текущем PR.

### 3. No speculative code (нет спекулятивного кода)

Сценарий не описан в acceptance — не реализуй молча. Спекулятивный код:
- добавляет поведение которое не тестировалось
- создаёт неявные зависимости
- усложняет ревью "а зачем это?"

Если сценарий очевидно нужен но не описан — подними вопрос через `/ask`. Не угадывай.

### 4. Assumptions explicit (явные допущения)

Если спека не покрывает граничный случай — не реализовывай молча. Сформулируй допущение явно и вызови `/ask`. Пример:

> "Спека не описывает поведение при пустом конфиге. Допущение: возвращать ErrInvalidConfig. Подтвердить?"

Молчаливое угадывание — это скрытый баг в понимании задачи.
