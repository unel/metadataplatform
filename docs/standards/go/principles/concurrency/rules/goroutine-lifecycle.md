# Жизненный цикл горутин

Каждая горутина должна иметь явный способ завершения. Утечка горутин — частый источник утечки памяти.

## Правило: shutdown path review при изменении горутинной структуры

Если в рамках одного code-fix изменяется горутинная структура сервиса (добавляется/удаляется горутина, меняется способ их координации, меняется порядок завершения) — ревьюер обязан пройти shutdown path целиком как отдельный шаг.

Что проверяется:
- Все горутины завершаются при получении сигнала остановки — нет горутин которые продолжают работать после `Stop()`
- `Stop()` идемпотентна — двойной вызов не паникует и не зависает
- Double-signal поведение: повторный сигнал (например второй SIGTERM) не вызывает панику
- Нет deadlock: горутины не ждут друг друга в цепочке которая никогда не разрешится
- WaitGroup / channel drain: все `wg.Wait()` и `<-done` имеют соответствующий `wg.Done()` / `close()` на каждом пути выполнения включая error paths

```go
// плохо — горутина никогда не завершится
go func() {
    for {
        process()
    }
}()

// хорошо — завершение через context
func startWorker(ctx context.Context, jobs <-chan Job) {
    go func() {
        for {
            select {
            case <-ctx.Done():
                return // завершаемся при отмене context
            case job, ok := <-jobs:
                if !ok {
                    return // завершаемся при закрытии канала
                }
                process(job)
            }
        }
    }()
}

// errgroup — горутины с обработкой ошибок
g, ctx := errgroup.WithContext(ctx)
for _, job := range jobs {
    job := job // захват переменной
    g.Go(func() error {
        return process(ctx, job)
    })
}
if err := g.Wait(); err != nil {
    return err
}
```
