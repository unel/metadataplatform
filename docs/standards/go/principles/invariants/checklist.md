# Чек-лист инвариантов: автор и ревьювер

## Автор — перед написанием кода

- [ ] Есть ли в этом блоке lifecycle, concurrency, shutdown или signal handling логика?
- [ ] Есть ли зависимость от порядка вызовов (A должен произойти до B)?
- [ ] Есть ли разделяемое состояние к которому обращаются несколько горутин?
- [ ] Операция предполагается идемпотентной — это зафиксировано?
- [ ] Каждый такой инвариант помечен `// inv:` комментарием?

## Автор — перед отправкой на ревью

- [ ] Все `// inv:` комментарии актуальны после изменений
- [ ] Если код переносился / реструктурировался — инварианты не нарушены в новом месте
- [ ] Shutdown path: cancel() вызывается до блокирующего ожидания (`<-done`, `wg.Wait()`, etc.)
- [ ] Нет блокирующих операций под удерживаемым мьютексом без явной пометки

## Ревьювер — при чтении lifecycle/concurrency/shutdown блоков

- [ ] Все инварианты в блоке задокументированы `// inv:` — нет неявных предположений
- [ ] Порядок вызовов в shutdown path: cancel() предшествует любому блокирующему ожиданию
- [ ] Если блок изменялся в этом PR — каждый инвариант проверен не нарушен
- [ ] Любое замечание на структуру или порядок в этих блоках — автоматически critical

## Ревьювер — обязательный вопрос

При каждом замечании к lifecycle/concurrency/shutdown блоку:

> **"Как предложенный фикс сохраняет инвариант?"**

Без ответа на этот вопрос замечание не выпускается — оно должно содержать явное объяснение того, что инвариант остаётся нарушенным или как он сохраняется.

## Red flags — немедленно critical

- cancel() вызывается после блокирующего `<-done` в shutdown ветке
- Закрытие канала не по принципу "закрывает отправитель"
- `sync.WaitGroup.Wait()` вызывается без гарантии что все `Add()` уже выполнены
- Мьютекс удерживается при блокирующем вызове (потенциальный дедлок)
- `context.CancelFunc` не вызывается при всех путях завершения функции
