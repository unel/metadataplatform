# Инварианты кода в Go

Инвариант — это утверждение, которое должно быть истинным в определённый момент выполнения программы. Нарушение инварианта — не баг в логике: это сбой контракта, который код давал самому себе.

В lifecycle, concurrency и shutdown блоках инварианты особенно опасны: они не видны в типах, не проверяются компилятором, и нарушаются только в конкретных сценариях (гонки, двойной сигнал, медленный shutdown). Именно поэтому они должны быть зафиксированы явно — в коде, рядом с тем местом, где они применяются.

## Категории инвариантов

### lifecycle

Порядок инициализации и завершения компонентов. Пример:

> "cancel() должен вызываться до любого блокирующего `<-done` в shutdown-ветках — иначе контекст не распространится в активные горутины."

### concurrency

Предположения о порядке доступа к разделяемому состоянию. Пример:

> "mu.Lock() должен удерживаться при чтении и записи active — нет исключений."

### shutdown

Порядок действий при завершении сервиса. Пример:

> "activeConns должен быть дренирован до закрытия listener — иначе часть соединений потеряет обработчик."

### signal handling

Идемпотентность и порядок при получении системных сигналов. Пример:

> "SIGTERM вызывает shutdown ровно один раз — повторный сигнал игнорируется."

### idempotency

Операции, которые безопасно вызывать многократно. Пример:

> "Close() идемпотентен — второй вызов возвращает ErrAlreadyClosed, не паникует."

### error classification

Разграничение ошибок по категориям с фиксированной семантикой. Пример:

> "ErrTimeout — всегда retryable, ErrInvalidConfig — никогда."

## Правило severity в lifecycle/concurrency блоках

В блоках категорий `lifecycle`, `concurrency`, `shutdown` и `signal handling` **нет категории "minor по стилю"**.

Любое замечание на структуру или порядок в этих блоках автоматически классифицируется как **critical** — потому что даже косметическое изменение порядка вызовов может нарушить инвариант и привести к дедлоку, утечке горутин или потере данных.

Ревьювер обязан спросить: "Как этот фикс сохраняет инвариант?" Без ответа на этот вопрос замечание не выпускается.

## Как документировать инварианты

Используй комментарий `// inv:` непосредственно перед блоком кода, к которому относится инвариант. Формат:

```go
// inv: <утверждение которое должно быть истинным>
// <объяснение почему это важно — одной строкой если неочевидно>
```
