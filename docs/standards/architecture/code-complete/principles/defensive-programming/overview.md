# Defensive Programming

McConnell разделяет два разных механизма защиты кода:

**Assertions** — проверки внутренних инвариантов (preconditions, postconditions). Работают в debug-режиме, отключаются в production. Цель: обнаружить ошибки разработчика как можно раньше.

**Error handling** — обработка ошибок которые могут возникнуть в production (неверный ввод, недоступный ресурс). Работает всегда.

**Barricade** — архитектурная граница между «грязными» и «чистыми» данными. Снаружи барьера — ненадёжные данные, требующие валидации. Внутри — доверенные данные, assertions уместны, error handling не нужен.
