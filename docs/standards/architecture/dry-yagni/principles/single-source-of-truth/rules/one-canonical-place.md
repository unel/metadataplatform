# Одно каноническое место

Каждое знание живёт в одном месте. Изменение требует правки только в одном файле.

**Почему:** дублированное знание рассинхронизируется. Правило меняется в одном месте — в другом остаётся старое. Система начинает противоречить сама себе.

## Границы применимости

Бизнес-правила, формулы, константы, конфигурация, схемы данных. Менее применимо к коду который выглядит похоже но выражает разные концепции.

## Если соблюдать рьяно

Всё выносится в центральные модули — появляются god-объекты с сотнями констант и правил. Зависимости между модулями растут.

## Если игнорировать

Лимит в 100 запросов в минуту прописан в трёх местах. При изменении обновляется два из трёх. Один сервис работает по старому правилу.

## Когда можно отступить

Когда похожие значения в разных местах выражают разные концепции — дублирование оправдано. Не каждое совпадение значений означает дублирование знания.

## Плохо

Одно правило в нескольких местах:

```
// auth service
maxLoginAttempts = 5

// api gateway
rateLimitPerUser = 5

// frontend
showLockoutAfter = 5
```

## Хорошо

Правило определено один раз, остальные берут из конфига:

```
// config
security.maxLoginAttempts = 5

// все остальные читают из конфига
attempts >= config.security.maxLoginAttempts -> lockout()
```
