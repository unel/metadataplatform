# Последний ответственный момент

Принимай решение когда промедление начинает стоить дороже чем ошибка. До этого момента — оставляй варианты открытыми.

**Почему:** чем раньше принято решение — тем меньше информации на руках. Раннее решение часто основано на предположениях которые окажутся неверными. Откладывание до последнего ответственного момента максимизирует информацию при минимальных потерях гибкости.

**Акцент YAGNI:** не строй механизм пока нет реальной потребности — появление потребности и есть последний ответственный момент.

## Границы применимости

Выбор хранилища, протокола, формата данных, архитектурного паттерна. Менее применимо к решениям с высокой стоимостью изменения (схема БД, публичный API).

## Если соблюдать рьяно

Откладывание решений до бесконечности — paralysis by analysis. Система не строится потому что "ещё не вся информация".

## Если игнорировать

Решение принято в день первый на основе предположений. Через месяц оказывается что требования другие — и решение нужно переделывать с большими потерями.

## Когда можно отступить

Решения с высокой стоимостью изменения (публичный API, схема БД, межсервисный протокол) — лучше принять осознанно раньше чем откладывать и потом менять дорого.

## Плохо

Решение принято до появления реальных требований:

```
// "нам точно понадобится поддержка нескольких БД"
// → сразу строим абстракцию поверх БД
interface Repository { ... }
class PostgresRepository { ... }
class MySQLRepository { ... }   // нет ни одного реального использования
```

## Хорошо

Абстракция появляется когда появилась вторая реализация:

```
// сначала — просто работаем с PostgreSQL напрямую
// когда понадобился MySQL — выделяем интерфейс по факту
interface Repository { ... }
```
