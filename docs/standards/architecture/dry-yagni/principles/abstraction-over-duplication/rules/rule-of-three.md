# Правило трёх

Первый раз — пишешь. Второй раз — терпишь. Третий раз — выделяешь абстракцию.

**Почему:** преждевременная абстракция по двум случаям часто неверна — неизвестно какое обобщение правильное. После трёх случаев паттерн виден достаточно чётко.

## Границы применимости

Логика с побочными эффектами, трансформации данных, паттерны запросов. Менее применимо к декларативному коду (конфиги, маппинги).

## Если соблюдать рьяно

Любые два похожих места немедленно абстрагируются. Появляются преждевременные обобщения которые потом мешают развитию.

## Если игнорировать

Одна и та же логика валидации в десяти обработчиках. Изменение правила — десять правок, девять из которых можно забыть.

## Когда можно отступить

Когда похожий код выражает разные концепции — совпадение случайное и абстракция была бы ложной связью.

## Плохо

Одна и та же проверка в трёх местах:

```
// handler A
if user.role != "admin" -> return forbidden()

// handler B
if user.role != "admin" -> return forbidden()

// handler C
if user.role != "admin" -> return forbidden()
```

## Хорошо

Проверка выделена один раз:

```
function requireAdmin(user) -> error:
    if user.role != "admin" -> return ErrForbidden
    return nil

// handler A, B, C
err = requireAdmin(user)
if err != nil -> return err
```
