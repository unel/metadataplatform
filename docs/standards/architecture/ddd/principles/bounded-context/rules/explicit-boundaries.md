# Явные границы

**Почему:** когда компонент знает о деталях другого — они связаны неявно. Изменение внутренностей одного ломает другой. Система становится монолитом по факту, даже если выглядит как набор компонентов.

## Границы применимости

Между архитектурными компонентами верхнего уровня (`cmd/store`, `cmd/spawner`, `cmd/api`, `workers/`, `hooks/`). Внутри одного компонента жёсткие зависимости допустимы.

## Если соблюдать рьяно

Каждое взаимодействие через явный интерфейс добавляет overhead: сериализация, сетевой вызов, версионирование протокола. Не нужно нарезать на контексты то что естественно живёт вместе.

## Если игнорировать

Worker начинает ходить в БД напрямую → теперь у него зависимость на postgres, конфиг, схему. Тестировать worker без БД невозможно. Изменение схемы БД → нужно обновить worker.

## Когда можно отступить

Shared kernel — небольшой набор общих типов (например, структура `Entity`) который используется несколькими контекстами. Важно: только типы, не логика.

## Плохо — worker знает о БД

```
// workers/hash/main.go
// worker напрямую читает из БД
db := postgres.Connect(os.Getenv("DB_URL"))
entity := db.Query("SELECT * FROM entities WHERE id = $1", entityId)

// обрабатывает...
hash := sha256(readFile(entity.Meta["path"]))

// и напрямую пишет в БД
db.Exec("INSERT INTO relations ...", entityId, hash)
```

Worker теперь знает о схеме БД, SQL, postgres-специфике.

## Хорошо — worker работает через протокол

```
// workers/hash/main.go
// получает данные через аргументы от spawner
entityId := flag.String("entity-id", "", "")

// если нужны детали — запрашивает через API (store socket)
entity := storeClient.Find(entityId)

// результат — JSONL в stdout, spawner читает и пишет в store
fmt.Println(`{"op":"upsert","type":"relation","data":{...}}`)
```

Границы: spawner → worker через CLI args + stdout JSONL. Worker ничего не знает о БД.
