# Расширяй, не изменяй

**Почему:** правка существующего кода — риск регрессий. Новый код, реализующий новое поведение, не затрагивает то что уже работает.

## Границы применимости

Точки вариативности — места где система должна поддерживать разные варианты поведения. Не нужно абстрагировать то что заведомо не будет меняться.

## Если соблюдать рьяно

Каждый класс обрастает интерфейсом даже если реализация одна и никогда не будет второй. Код утопает в абстракциях без реальной причины.

## Если игнорировать

Добавление нового формата/типа/варианта — это правка в 5 местах, каждое из которых можно забыть. Логика рассыпана по всему коду.

## Когда можно отступить

Если точка расширения появилась один раз и явно не появится снова — не надо абстрагировать заранее. OCP применяется к известным точкам вариативности, а не ко всему подряд.

## Плохо

Добавление нового воркера требует правки существующего dispatcher:

```
function dispatch(job):
  if job.worker == "hash.sha256":
    return HashWorker.run(job)
  else if job.worker == "scanner":
    return ScannerWorker.run(job)
  else if job.worker == "stash":    // добавляем — правим существующий код
    return StashWorker.run(job)
  else:
    throw "unknown worker"
```

## Хорошо

Новый воркер — новый файл, dispatcher не трогаем:

```
interface Worker:
  run(job: Job) -> Result

// существующий код не меняется
class HashWorker implements Worker: ...
class ScannerWorker implements Worker: ...

// добавляем нового — только новый файл
class StashWorker implements Worker:
  run(job): ...

// dispatcher работает с интерфейсом
function dispatch(worker: Worker, job: Job):
  return worker.run(job)
```

В проекте: воркеры — внешние бинари. Spawner матчит job к файлу `workers/<type>.<subtype>`. Добавление нового воркера = новый файл в `workers/`, spawner не меняется — это OCP на уровне файловой системы.
