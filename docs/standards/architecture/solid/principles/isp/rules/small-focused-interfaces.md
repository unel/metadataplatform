# Маленькие целевые интерфейсы

**Почему:** большой интерфейс создаёт ненужные зависимости. Изменение метода которым клиент не пользуется вынуждает клиента перекомпилироваться/обновляться. Тестирование усложняется — моки нужно реализовывать полностью.

## Границы применимости

Интерфейсы которые реализуются несколькими классами или принимаются в нескольких местах. Единственная реализация с единственным потребителем — не повод дробить.

## Если соблюдать рьяно

Интерфейсы дробятся до одного метода везде. Код перегружен однометодными интерфейсами, которые никогда не комбинируются по-разному.

## Если игнорировать

«Толстые» интерфейсы с десятками методов. Реализации заполнены заглушками. Моки для тестов огромны. Каждое изменение интерфейса затрагивает всю кодовую базу.

## Когда можно отступить

Если интерфейс — публичный API внешнего контракта (REST, RPC), его нельзя просто так раздробить. Тут ISP применяется на уровне endpoint'ов, не методов одного класса.

## Плохо

Функции принимают полный интерфейс, используя один метод:

```
interface EntityStore:
  findEntity(id) -> Entity
  saveEntity(entity) -> void
  deleteEntity(id) -> void
  findRelation(id) -> Relation
  saveRelation(relation) -> void
  queryEntities(filter) -> []Entity
  queryRelations(filter) -> []Relation
  // ... ещё 10 методов

// использует только findEntity и queryEntities
function buildIndex(store: EntityStore):
  entities = store.queryEntities({})
  for e in entities:
    store.findEntity(e.id) // зависит от огромного интерфейса ради 2 методов
```

## Хорошо

Интерфейс по роли, функция принимает только то что нужно:

```
interface EntityReader:
  findEntity(id) -> Entity
  queryEntities(filter) -> []Entity

interface EntityWriter:
  saveEntity(entity) -> void
  deleteEntity(id) -> void

// зависит только от того что использует
function buildIndex(reader: EntityReader):
  entities = reader.queryEntities({})
  ...
```

В Go: определяй интерфейс рядом с потребителем, не рядом с реализацией — это и есть ISP в идиоматичном Go.
