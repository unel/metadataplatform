# Одна причина для изменения

**Почему:** когда модуль отвечает за несколько вещей, изменение одной из них ломает или запутывает остальные. Код становится хрупким — правка в одном месте неожиданно затрагивает другое.

## Границы применимости

Классы, модули, функции, сервисы. Не применяй механически к каждой функции — иначе раздробишь естественно связанную логику на бессмысленные осколки.

## Если соблюдать рьяно

Модули дробятся до тривиальности. Каждый файл содержит одну функцию, навигация становится невозможной. Это «раздробление ради принципа» — хуже нарушения.

## Если игнорировать

Появляются «god objects» — модули на тысячи строк, которые делают всё. Любое изменение требует понимания всего модуля. Тесты невозможно изолировать.

## Когда можно отступить

Если разделение требует сложной координации между частями — возможно это действительно одна ответственность. Граница ответственности не всегда очевидна.

## Плохо

Один модуль смешивает три ответственности: вычисление, форматирование, запись:

```
module Report:
  function generate(data):
    // считаем метрики (бизнес-логика)
    total = sum(data.values)
    avg = total / len(data)

    // форматируем в HTML (отображение)
    html = "<table>..." + total + avg + "...</table>"

    // пишем в файл (IO)
    file.write("report.html", html)
    return html
```

Изменение формата вывода → трогаем модуль с бизнес-логикой.
Изменение хранилища → трогаем модуль с форматированием.

## Хорошо

Три отдельных модуля с чёткими границами:

```
module ReportCalculator:
  function calculate(data) -> ReportData:
    return {total: sum(data), avg: avg(data)}

module ReportFormatter:
  function toHTML(report: ReportData) -> string:
    return "<table>..." + report.total + "...</table>"

module ReportWriter:
  function save(path, content):
    file.write(path, content)
```

Изменение формата → только `ReportFormatter`. Изменение хранилища → только `ReportWriter`.
