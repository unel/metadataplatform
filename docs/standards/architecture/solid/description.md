# SOLID

Пять принципов объектно-ориентированного проектирования, применимых к любому стеку.

## Принципы

**S — Single Responsibility**
У каждого модуля должна быть ровно одна причина для изменения.
Признак нарушения: модуль приходится менять по двум разным причинам — значит он совмещает две ответственности.

**O — Open/Closed**
Открыт для расширения, закрыт для изменения.
Признак нарушения: добавление новой фичи требует правки существующего кода.

**L — Liskov Substitution**
Подтип должен быть заменяем базовым типом без изменения поведения.
Признак нарушения: наследник переопределяет поведение так что контракт базового типа нарушается.

**I — Interface Segregation**
Много специализированных интерфейсов лучше одного универсального.
Признак нарушения: реализация интерфейса содержит методы которые не нужны.

**D — Dependency Inversion**
Зависимость от абстракций, не от конкретных реализаций.
Признак нарушения: высокоуровневый модуль напрямую создаёт низкоуровневые объекты.

## Применение в проекте

- **Go**: интерфейсы для зависимостей (store, spawner), инъекция через конструктор
- **SvelteKit**: компоненты зависят от props/stores, не от конкретных API вызовов
- **CSS**: компоненты зависят от design tokens, не от конкретных значений
