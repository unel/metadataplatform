# Держи комментарии актуальными

Устаревший комментарий хуже чем его отсутствие — он активно вводит в заблуждение.

**Почему:** код меняется, комментарии — нет. Читатель доверяет комментарию и строит неверную модель. Ложный комментарий опаснее отсутствующего.

## Границы применимости

Весь код. Особенно критично для комментариев описывающих алгоритм, порядок операций или бизнес-правила.

## Если соблюдать рьяно

Обновление комментариев при каждом изменении кода — это дополнительная работа которую легко забыть. Иногда проще удалить комментарий чем поддерживать его актуальность.

## Если игнорировать

Комментарий говорит "кэш обновляется каждые 5 минут" — а код обновляет каждые 30. Разработчик настраивает систему под неверные ожидания.

## Когда можно отступить

Комментарии в истории git (commit messages) — они фиксируют состояние на момент изменения и не должны обновляться.

## Плохо

Комментарий описывает старое поведение:

```
// делим на количество воркеров (всегда 4)
limit = total / workerCount
```

Но `workerCount` теперь динамический и может быть любым числом.

## Хорошо

Комментарий обновлён вместе с кодом — или удалён если код говорит сам за себя:

```
limit = total / workerCount
```
