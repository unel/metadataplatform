# Сначала улучши код — потом комментируй

Если хочется добавить комментарий — сначала попробуй переименовать, выделить функцию или упростить структуру. Комментарий — последнее средство.

**Почему:** комментарий не проверяется компилятором и устаревает. Хорошее имя или маленькая функция — самодокументирующийся код который всегда актуален.

## Границы применимости

Любой код где появляется желание написать "это делает X". Менее применимо к публичному API где документационные комментарии обязательны по соглашению.

## Если соблюдать рьяно

Погоня за самодокументирующимся кодом приводит к искусственным абстракциям. Выделяется функция ради единственного вызова только чтобы не писать комментарий.

## Если игнорировать

Код обрастает комментариями-костылями вместо того чтобы становиться лучше. Комментарии скрывают проблему а не решают её.

## Когда можно отступить

Алгоритмы со сложной математикой, хаки для обхода известных багов платформы, неочевидные бизнес-правила — там комментарий необходим даже если код чистый.

## Плохо

Комментарий скрывает плохое имя:

```
// проверяем активен ли пользователь и не истёк ли его токен
if u.a && !u.te {
    ...
}
```

## Хорошо

Код говорит сам за себя:

```
if user.isActive() && !user.tokenExpired() {
    ...
}
```
