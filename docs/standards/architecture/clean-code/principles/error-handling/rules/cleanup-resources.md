# Освобождай ресурсы при ошибке

Ресурсы (файлы, соединения, транзакции) должны освобождаться при любом исходе — в том числе при ошибке.

**Почему:** утечка ресурсов накапливается незаметно. Одна необработанная ошибка не ломает систему сразу — но под нагрузкой приводит к исчерпанию соединений, дескрипторов, памяти.

## Границы применимости

Любой код который открывает файлы, соединения, транзакции, lock'и. Особенно критично в долгоживущих сервисах и под нагрузкой.

## Если соблюдать рьяно

Избыточные очистки там где ресурс управляется внешним контекстом (например connection pool). Двойное закрытие может само по себе вызвать ошибку.

## Если игнорировать

Утечки соединений с БД. Незакрытые файловые дескрипторы. Незафиксированные и незаоткаченные транзакции. Всё это проявляется под нагрузкой как деградация производительности.

## Когда можно отступить

Ресурсы управляемые внешним контекстом (request context, connection pool) — там cleanup выполняется автоматически при завершении контекста.

## Плохо

Ресурс закрывается только в конце успешного пути:

```
f = open(path)
if err != nil -> return err

process(f)
f.close()  // не вызовется если process вернул ошибку
```

## Хорошо

Освобождение регистрируется сразу после получения ресурса:

```
f = open(path)
if err != nil -> return error("open {path}", err)

defer f.close()  // вызовется при любом исходе

return process(f)
```
