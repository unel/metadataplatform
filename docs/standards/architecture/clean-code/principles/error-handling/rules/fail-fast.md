# Проверяй ошибки сразу

Ошибка проверяется сразу после вызова — не откладывается на потом.

**Почему:** отложенная проверка увеличивает расстояние между причиной и обнаружением. Чем позже проверяешь — тем труднее понять что пошло не так и в каком состоянии система.

## Границы применимости

Последовательные операции где каждая зависит от предыдущей. В параллельном коде паттерн может отличаться — ошибки собираются и обрабатываются после завершения всех горутин/промисов.

## Если соблюдать рьяно

В параллельном коде это неприменимо — нельзя проверить ошибку горутины "сразу". Принуждение к fail-fast в таких случаях усложняет код без пользы.

## Если игнорировать

Следующий шаг выполняется с результатом упавшего предыдущего. Система продолжает работу в частично сломанном состоянии. Ошибка обнаруживается поздно и далеко от источника.

## Когда можно отступить

- Параллельное выполнение: ошибки собираются после завершения всех операций
- Валидация форм: собираем все ошибки сразу чтобы показать пользователю

## Плохо

Несколько вызовов подряд, проверка — в конце:

```
result1, err1 = stepOne()
result2, err2 = stepTwo(result1)
result3, err3 = stepThree(result2)

if err1 != nil -> return err1
if err2 != nil -> return err2
if err3 != nil -> return err3
```

## Хорошо

Каждый вызов проверяется сразу:

```
result1, err = stepOne()
if err != nil -> return error("step one", err)

result2, err = stepTwo(result1)
if err != nil -> return error("step two", err)

result3, err = stepThree(result2)
if err != nil -> return error("step three", err)
```
