# Не игнорируй ошибки

Каждая возвращаемая ошибка должна быть обработана — явно проигнорировать можно только с комментарием почему.

**Почему:** проигнорированная ошибка — это бомба замедленного действия. Программа продолжает работу в неопределённом состоянии, и симптомы появятся далеко от причины.

## Границы применимости

Все вызовы функций которые возвращают ошибку или могут завершиться неудачей. Особенно критично для операций с I/O, сетью, БД.

## Если соблюдать рьяно

Код обрастает обработчиками даже для операций которые физически не могут упасть. Читаемость падает, сигнал-шум ухудшается.

## Если игнорировать

Программа продолжает работу после неудачной записи в файл, неудачного удаления, битого парсинга. Баги проявляются в совершенно другом месте через неопределённое время.

## Когда можно отступить

- Операции очистки в deferred-блоках: ошибка при закрытии файла редко требует обработки если основная операция уже завершена
- Best-effort операции (логирование, метрики): если отправка метрики упала — программа должна продолжить работу

## Плохо

Вызов без проверки результата:

```
delete(tmpFile)
parse(data, result)
```

## Хорошо

Каждый вызов проверяется, ошибка обрабатывается или пробрасывается:

```
err = delete(tmpFile)
if err != nil -> return error("cleanup temp file", err)

err = parse(data, result)
if err != nil -> return error("parse response", err)
```
