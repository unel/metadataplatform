# Имена раскрывающие намерение

Имя должно говорить зачем переменная существует и как используется.

**Почему:** код читают чаще чем пишут. Непонятное имя заставляет держать контекст в голове и мысленно декодировать каждое обращение к переменной.

## Границы применимости

Все публичные и пакетные переменные, параметры функций, поля структур. Локальные переменные в очень коротких блоках (2-3 строки) допускают краткость.

## Если соблюдать рьяно

Имена становятся многословными и неудобными: `elapsedTimeSinceLastSuccessfulSyncInSeconds`. Читатель тратит время на разбор имени вместо понимания логики.

## Если игнорировать

Код превращается в шифр — `d`, `x`, `tmp2`. Новый разработчик не может понять намерение без запуска кода и отладчика.

## Когда можно отступить

- Счётчики циклов: `i`, `j` — общепринятое соглашение
- Параметры однострочных лямбд: `x` в `map(x -> x * 2)`
- Математические формулы: `a`, `b`, `c` если это действительно коэффициенты

## Плохо

Имя ничего не говорит о смысле — нужен комментарий:

```
d = 0  // elapsed time in days
```

## Хорошо

Имя само является документацией:

```
elapsedTimeInDays = 0
```
