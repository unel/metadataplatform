# Маленький размер

Функция должна умещаться на экране, идеал — 5-10 строк.

**Почему:** длинная функция — признак что она делает слишком много. Читатель вынужден держать в голове весь контекст от начала до конца. Маленькая функция легко тестируется, легко именуется, легко понимается.

## Границы применимости

Бизнес-логика, сервисный слой, обработчики. Менее применимо к декларативным конфигурациям и инициализации где длина оправдана перечислением.

## Если соблюдать рьяно

Код разбивается на сотни однострочных функций. Навигация становится невозможной — чтобы понять одно действие нужно прыгать по десяти файлам. Абстракции ради абстракций.

## Если игнорировать

Функции на 200 строк с вложенными условиями и циклами. Невозможно написать unit-тест. Любое изменение ломает что-то неожиданное.

## Когда можно отступить

- Switch/match по большому числу случаев — допустимо если каждый случай тривиален
- Инициализация конфига или DI-контейнера — длина оправдана перечислением зависимостей
- Генерированный код

## Плохо

Одна функция содержит несколько логических блоков:

```
function processOrder(order):
    // валидация — 10 строк
    // расчёт скидки — 15 строк
    // обновление склада — 12 строк
    // отправка уведомления — 8 строк
```

## Хорошо

Функция верхнего уровня — только последовательность шагов:

```
function processOrder(order):
    err = validateOrder(order)
    if err != nil -> return err

    order.total = applyDiscount(order)

    err = updateInventory(order)
    if err != nil -> return err

    return notifyCustomer(order)
```
