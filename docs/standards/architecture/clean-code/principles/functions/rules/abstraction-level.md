# Один уровень абстракции

Все операции внутри функции должны находиться на одном уровне абстракции.

**Уровни абстракции** — это "дистанция от деталей реализации":
- **Высокий**: бизнес-понятия — `parseRequest()`, `saveUser()`, `sendWelcomeEmail()`
- **Средний**: прикладная логика — `user.validate()`, `repo.find(id)`
- **Низкий**: детали реализации — обрезка пробелов, побайтовая запись, форматирование строк

**Почему:** когда уровни смешаны, читатель не понимает что здесь главное, а что деталь. Функции начинают "разрастаться" — сначала одна деталь, потом другая.

## Границы применимости

Бизнес-логика и сервисный слой. В низкоуровневых утилитах смешение уровней иногда неизбежно.

## Если соблюдать рьяно

Каждая строка выносится в отдельную функцию. Появляются функции-обёртки ради единственного вызова — `function trimName(user): user.name = trim(user.name)`. Навигация усложняется без реальной пользы.

## Если игнорировать

Высокоуровневые функции обрастают деталями. Невозможно понять бизнес-логику не разбираясь в деталях реализации. Функции постепенно растут без видимой причины.

## Когда можно отступить

В тестах: setup-код часто смешивает уровни для наглядности. В скриптах и утилитах: короткий скрипт без абстракций читается лучше чем разбитый на функции ради принципа.

## Плохо

Высокоуровневая логика смешана с деталью реализации:

```
function handleRequest(request):
    user = parseRequest(request)        // высокий уровень
    user.name = trim(user.name)         // низкий уровень
    saveUser(user)                      // высокий уровень
    sendWelcomeEmail(user)              // высокий уровень
```

## Хорошо

Все операции на одном уровне, деталь скрыта за именем:

```
function handleRequest(request):
    user = parseRequest(request)
    normalizeUser(user)
    saveUser(user)
    sendWelcomeEmail(user)

function normalizeUser(user):
    user.name = trim(user.name)
    user.email = lowercase(user.email)
```
