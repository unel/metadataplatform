# Command/Query разделение

Функция либо выполняет команду (меняет состояние), либо возвращает результат запроса — не оба сразу.

**Почему:** смешение делает вызов двусмысленным. Читатель не знает — это вопрос или действие? Нужно заглядывать в реализацию.

## Границы применимости

Сервисный слой, репозитории, доменные объекты. Менее применимо к builder-паттерну где цепочка методов меняет состояние и возвращает себя.

## Если соблюдать рьяно

Атомарные операции типа "найди или создай" становятся неудобными — нужно два вызова вместо одного. Транзакционные сценарии усложняются.

## Если игнорировать

`if (set("key", "value"))` — это проверка или присвоение? Вызывающий код полагается на побочный эффект внутри запроса. Тестировать такие функции крайне неудобно.

## Когда можно отступить

- Builder-паттерн: `query.where(x).limit(10).execute()` — каждый метод меняет состояние и возвращает себя
- "Find or create" операции: атомарность важнее разделения
- Стековые операции: `stack.pop()` — по природе и команда и запрос

## Плохо

Функция и меняет состояние, и возвращает результат:

```
function set(attr, value) -> bool:
    if exists(attr):
        attributes[attr] = value
        return true
    return false

if set("username", "bob"):  // это проверка или присвоение?
    ...
```

## Хорошо

Команда и запрос разделены:

```
function set(attr, value):
    attributes[attr] = value

function exists(attr) -> bool:
    return attr in attributes

if exists("username"):
    set("username", "bob")
```
