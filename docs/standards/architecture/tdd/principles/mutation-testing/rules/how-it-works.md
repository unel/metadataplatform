# Как работает mutation testing

## Типы мутаций

Инструмент вносит одно изменение за раз и запускает тесты:

| Мутация | Оригинал | Мутант |
|---------|----------|--------|
| Arithmetic | `a + b` | `a - b` |
| Comparison | `x > 0` | `x >= 0`, `x < 0` |
| Logical | `a && b` | `a \|\| b` |
| Boundary | `i < len` | `i <= len` |
| Return | `return result` | `return null` |
| Negation | `if valid` | `if !valid` |

## Интерпретация результатов

**Мутант убит** — тест упал. Хорошо, тест защищает это поведение.

**Мутант выжил** — тест не упал. Варианты:
- Поведение не покрыто тестом → добавить тест
- Эквивалентная мутация — изменение не меняет наблюдаемого поведения → игнорировать

**Эквивалентная мутация** — например, `i <= len-1` и `i < len` идентичны по смыслу. Инструмент не может их отличить автоматически.

## Пример

```
// код
function applyFilter(entities, filter):
  if filter.type == null:
    return entities
  return entities.filter(e -> e.type == filter.type)

// тест
test("applyFilter: фильтрует по типу"):
  result = applyFilter([{type:"file"}, {type:"job"}], {type:"file"})
  assert result == [{type:"file"}]
```

Мутант: меняем `== null` на `!= null` — возвращаем entities только когда filter.type задан.

Тест не покрывает случай `filter.type == null` → мутант выживает → нужен тест:

```
test("applyFilter: без фильтра возвращает все"):
  all = [{type:"file"}, {type:"job"}]
  assert applyFilter(all, {}) == all
```

## Когда запускать

Mutation testing медленный — на большой кодовой базе может занять часы. Оптимальные точки:

- **По требованию** — перед релизом критических модулей
- **На PR** — только для изменённых файлов (Stryker поддерживает `--since`)
- **По расписанию** — ночной прогон в CI для ключевых пакетов

Не запускай на каждом коммите — это убьёт скорость разработки.

## Инструменты в проекте

**Go** — `gremlins` (активнее поддерживается):
```bash
gremlins unleash ./internal/...
```

**TypeScript/SvelteKit** — Stryker:
```bash
npx stryker run
```

Конфиг Stryker — исключай генерированный код и тривиальные файлы:
```json
{
  "ignore": ["**/*.gen.ts", "**/index.ts"],
  "thresholds": {"break": 70}
}
```

## Mutation score — сколько достаточно

Нет универсального ответа. Ориентиры:

- **Критическая бизнес-логика** (парсинг, фильтрация, расчёты) — 80%+
- **Инфраструктурный код** (адаптеры, конфиг) — не гнаться, 50-60% нормально
- **Генерированный код** (sqlc, OpenAPI) — исключить совсем

Высокий coverage при низком mutation score — признак тестов которые выполняют код но не проверяют поведение.
