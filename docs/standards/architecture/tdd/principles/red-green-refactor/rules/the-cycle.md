# Цикл Red → Green → Refactor

**Почему:** написание кода до теста лишает TDD главного — обратной связи о дизайне. Если код сложно тестировать, это сигнал что у него плохой дизайн. Тест написанный заранее вскрывает это до того как код зарастёт зависимостями.

## Границы применимости

Новая функциональность и исправление багов (баг-фикс начинается с теста воспроизводящего баг). Исследовательский код (spikes) — можно без TDD, но результат нужно переписать с тестами.

## Если соблюдать рьяно

Каждая строчка кода за тестом — замедляет на начальном этапе, особенно для очевидной логики. Тривиальный код (геттеры, константы) тестировать не нужно.

## Если игнорировать

Тесты пишутся после — они проверяют реализацию, а не поведение. Покрытие высокое, уверенность низкая. Рефакторинг страшен — непонятно что сломается.

## Когда можно отступить

Прототипирование архитектурных решений (spike): сначала понять как это должно работать, потом выбросить и написать с тестами.

## Плохо — тест после кода, проверяет реализацию

```
// сначала написали:
function parseFilter(input):
  parts = input.split(",")
  result = {}
  for part in parts:
    [k, v] = part.split("=")
    result[k] = v
  return result

// потом написали тест под реализацию:
test("parseFilter splits by comma then equals"):
  assert parseFilter("type=file,subtype=video") == {type: "file", subtype: "video"}
  // тест знает про split — это деталь реализации
```

## Хорошо — тест сначала, описывает поведение

```
// RED: пишем тест до кода
test("parseFilter: извлекает тип и подтип из строки"):
  result = parseFilter("type=file,subtype=video")
  assert result.type == "file"
  assert result.subtype == "video"

test("parseFilter: пустая строка → пустой фильтр"):
  assert parseFilter("") == {}

test("parseFilter: неизвестный ключ игнорируется"):
  result = parseFilter("type=file,unknown=value")
  assert result.type == "file"
  assert result.unknown == undefined

// GREEN: минимальный код который проходит тесты
// REFACTOR: улучшаем код не трогая тесты
```
