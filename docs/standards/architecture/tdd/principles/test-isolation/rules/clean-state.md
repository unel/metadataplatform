# Чистое состояние

**Почему:** зависимый от порядка тест — это бомба замедленного действия. Сначала он работает, потом кто-то добавляет тест выше по файлу и всё ломается. Диагностировать такие падения крайне сложно.

## Границы применимости

Все тесты. Особенно критично для интеграционных — реальная БД накапливает данные между тестами если их не чистить.

## Если соблюдать рьяно

Каждый тест поднимает и сносит инфраструктуру — тест suite работает часами. Баланс: очищай данные, не инфраструктуру. Контейнер с БД можно держать один на весь suite, откатывая транзакцию после каждого теста.

## Если игнорировать

Flaky тесты. Тесты которые проходят локально но падают в CI. Тест suite который нельзя запустить в параллельном режиме.

## Когда можно отступить

Read-only тесты (только SELECT) могут делить состояние — они его не меняют. Фикстуры только для чтения можно создать один раз на весь suite.

## Плохо — тесты зависят от общего состояния

```
// тест 1 создаёт запись
test("create entity"):
  store.create({id: "1", type: "file"})
  assert store.count() == 1

// тест 2 предполагает что запись от теста 1 существует
test("find entity"):
  result = store.find("1")  // упадёт если запустить без теста 1
  assert result.type == "file"

// тест 3 падает потому что тест 1 уже создал запись
test("create another entity"):
  store.create({id: "2", type: "job"})
  assert store.count() == 1  // упадёт — в БД уже 2 записи от теста 1
```

## Хорошо — каждый тест независим

```
// каждый тест сам создаёт нужные данные и чистит после себя
setup:
  tx = db.beginTransaction()

teardown:
  tx.rollback()  // все изменения теста откатываются

test("find entity"):
  store.create({id: "abc", type: "file"})  // создаём сами
  result = store.find("abc")
  assert result.type == "file"
  // после теста — rollback, БД чистая

test("count entities"):
  store.create({id: "x", type: "file"})
  store.create({id: "y", type: "job"})
  assert store.count() == 2  // гарантированно 2, не N+2
```
