# Правило: проверяй attrs, не msg

## Формулировка

Тест, проверяющий наличие данных в логе, должен обращаться к структурированным полям (attrs), а не искать подстроку в текстовом сообщении (msg).

## Почему это важно

`waitForMsg("conn_id")` ищет строку `"conn_id"` в поле msg записи лога.

Если логгер пишет `conn_id` как структурированный атрибут — в msg этой строки нет.

Результат: функция возвращает `false` на каждом вызове. Тест "проходит" только потому что по истечении таймаута код идёт дальше без реальной проверки. Это эквивалент `time.Sleep`.

## Неправильно

```go
// waitForMsg ищет в поле msg — conn_id там нет, оно в attrs
ok := waitForMsg("conn_id")
assert.True(t, ok, "expected conn_id in log")
```

## Правильно

```go
// waitForAttr ищет по ключу в структурированных полях
ok := waitForAttr("conn_id", expectedConnID)
assert.True(t, ok, "expected log attr conn_id=%q", expectedConnID)
```

## Grep-конвенция при обнаружении паттерна

Если ревью или отладка выявили `waitForMsg` с именем атрибута как аргументом — это паттерн, не точечная ошибка.

**Перед закрытием замечания:**

1. Сформулируй паттерн словами: "вызов waitForMsg где аргумент — имя структурированного атрибута"
2. Grep по пакету/файлу — не по буквальной строке, а по сигнатуре паттерна
3. Убедись что других экземпляров нет
4. Только после этого закрывай замечание

Пример grep:
```bash
grep -n "waitForMsg(" ./internal/store/...
```

Если нашёл дополнительные вхождения — фикс распространяется на все.

## Применимость

Правило не привязано к конкретному логгеру. Применяется везде где:
- логгер пишет структурированные поля отдельно от текстового сообщения
- тест проверяет содержимое лога
- есть хелпер для ожидания записи в логе
