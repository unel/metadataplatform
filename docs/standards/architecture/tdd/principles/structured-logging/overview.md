# Structured Logging: тесты должны проверять attrs, не msg

## Два слоя структурированного лога

Структурированный лог — это не строка. Это запись с двумя принципиально разными слоями:

**msg** — человекочитаемый шаблон события.
- Может меняться при локализации, рефакторинге, уточнении формулировки.
- Не является контрактом.
- Пример: `"connection established"`, `"store error"`, `"request processed"`

**attrs (structured fields)** — именованные поля с типизированными значениями.
- Это контракт между компонентами системы.
- Не меняются без явного изменения интерфейса.
- Пример: `conn_id="abc-123"`, `duration_ms=42`, `error="timeout"`

## Принцип

> Тест должен проверять attrs — не подстроку в msg.

Причина: msg предназначен для людей, attrs — для машин. Тест — это машина.

Если тест проверяет подстроку в msg, он:
1. Ломается при любом изменении формулировки (ложный red)
2. Пропускает реальные ошибки когда нужное поле пишется в attrs, а не в msg (ложный green)

## Формулировка через принцип, не через конкретный логгер

Этот стандарт не привязан к `slog`, `zap`, `logrus` или любому другому логгеру.

Принцип: **проверяй структурированные поля, не текстовое представление**.

Если логгер не даёт доступа к структурированным полям из теста — это сигнал что нужен тестовый хелпер или замена логгера на интерфейс с наблюдаемым выводом.

## Примеры на Go с slog

### Неправильно: проверка подстроки в msg

```go
// Ожидание: лог содержит conn_id
// Реальность: conn_id пишется в attrs — waitForMsg всегда возвращает false.
// Тест работает как time.Sleep — проходит по таймауту, не по условию.

ok := waitForMsg("conn_id")
if !ok {
    t.Fatal("expected conn_id in log")
}
```

### Правильно: проверка структурированного поля

```go
// Проверяем что в логе появилась запись с атрибутом conn_id = expectedID

ok := waitForAttr("conn_id", expectedID)
if !ok {
    t.Fatalf("expected log attr conn_id=%q", expectedID)
}
```

### Как устроен тестовый хелпер

```go
// logEntry — одна запись из буфера логгера в тесте
type logEntry struct {
    Msg   string
    Attrs map[string]any
}

// waitForAttr ждёт появления записи с заданным ключом и значением в attrs
func waitForAttr(key string, value any) bool {
    deadline := time.Now().Add(timeout)
    for time.Now().Before(deadline) {
        for _, entry := range logBuffer.Entries() {
            if entry.Attrs[key] == value {
                return true
            }
        }
        time.Sleep(10 * time.Millisecond)
    }
    return false
}
```

## Следствия

- Тестовый логгер должен предоставлять доступ к attrs отдельно от msg.
- Хелперы типа `waitForMsg(string)` — красный флаг. Замени на `waitForAttr(key, value)`.
- Если нужно проверить сообщение — проверяй его явно как отдельное поле, не через поиск подстроки.
