# Базовый план: 06-retro-recall

## 0. Prime Directive и review предыдущего ретро

**Prime Directive** (Norm Kerth): каждый агент действовал из лучших побуждений при имеющихся данных, инструментах и возможностях. Критика направлена на систему и процесс — не на конкретного агента.

**Review предыдущего RETRO.md**: если в директории фичи (или смежных фич) есть предыдущий `RETRO.md` с action items — прочитай его. Для каждого item: выполнен / не выполнен / устарел. Устаревшие — явно списать, не тащить вперёд.

## 1. Определение участников

Прочитай директорию фичи — определи кто участвовал:

- Есть `spec.md`, `acceptance.md`, `spec-fix-*`, `acceptance-fix-*` → **Танк**
- Есть код фичи, `code-fix-*` → **Ада**
- Есть `*-review-*` → **Гримм**
- Есть тест-файлы, `test-fix-*`, `test-run-report` → **Кроули** и **Азирафаль**
- **Харли** — всегда, как оркестратор

## 2. Параллельный recall

Спауни всех участников одновременно с `mode: bypassPermissions`.

Каждый агент:
1. Читает свои артефакты фичи
2. Читает `notes-user.md` и `complaints-user.md` если существуют
3. **Дополняет** (append) свои `notes-<агент>.md` и `complaints-<агент>.md` — не перезаписывает

### Формат `notes-<агент>.md`

Одна строка = одно наблюдение с тегом:

```
[rework] что пришлось переделывать и почему
[friction] что замедляло работу
[miss] что забыли или пропустили
[propose] предложение по улучшению процесса или инструментов
[doc] нюанс который стоит задокументировать
[whatever] не вписывается — разберёмся на ретро
```

### Формат `complaints-<агент>.md`

Сырые жалобы без тегов и без фильтра. Что раздражало, что казалось бессмысленным, на кого злился.

## 3. Формирование отчёта

После завершения всех агентов:
- Сводка: какие файлы обновлены, сколько наблюдений добавил каждый
- Незаполненные файлы если агент ничего не нашёл

## Движение вперёд и назад

**Передаём:** report-NNN.md (список обновлённых файлов)
