# Базовый чек-лист: 03-tests-write

## Контекст

- [ ] Acceptance прочитан (актуальная версия — последний report из `02-acceptance/`)
- [ ] Spec прочитана для понимания архитектурного контекста
- [ ] `PROJECT.md` прочитан — стек, соглашения по тестированию

## Структура файлов

- [ ] Тесты happy path размещены в `tests/happy/`
- [ ] Тесты adversarial размещены в `tests/adversarial/`
- [ ] Каждый тест-модуль ≤ 150 строк (без комментариев); если больше — разбит на файлы по группам

## FIRST принципы

- [ ] **Fast** — медленные зависимости не вызываются напрямую в unit тестах
- [ ] **Isolated** — каждый тест независим, нет общего mutable state
- [ ] **Repeatable** — нет `time.Now()` без инъекции, нет `rand` без фиксированного seed, нет сетевых вызовов
- [ ] **Self-validating** — тест сам определяет pass/fail
- [ ] **Timely** — тесты написаны к acceptance, до финального кода

## Поведение, не реализация

- [ ] Тесты проверяют наблюдаемое поведение — не внутренние вызовы методов
- [ ] Тесты стабильны к рефакторингу — не сломаются если переименовать внутренний метод

## Именование

- [ ] Имя каждого теста = сценарий + ожидаемый результат (читается как документация)

## Own setup

- [ ] Каждый тест сам подготавливает данные — нет зависимости от других тестов
- [ ] Использует `t.Cleanup` / `defer` для очистки состояния

## Покрытие acceptance

- [ ] Каждый happy path сценарий из acceptance покрыт тестом (Азирафаль)
- [ ] Каждый failure mode из acceptance покрыт тестом (Кроули)
- [ ] Граничные условия покрыты (Кроули)
- [ ] Поведение при недоступности зависимостей покрыто (Кроули)
- [ ] НФТ которые можно проверить тестом — покрыты

## Red run (обязательно)

- [ ] Тесты запущены до реализации — `verify: тест падает без реализации (Red run)`
- [ ] Если тест проходит без реализации — это false positive, разобрано до продолжения

## Прогон

- [ ] Все написанные тесты проходят
- [ ] Нет игнорируемых / закомментированных тестов без объяснения

## Маппинг

- [ ] В report: каждый тест привязан к сценарию acceptance
- [ ] Непокрытые сценарии если есть — явно перечислены с объяснением

## Движение дальше

- [ ] Все сценарии acceptance покрыты
- [ ] Тесты проходят — report написан, 03-tests/02-review может начинать
- [ ] Если acceptance неполный или противоречив — `clarification`, откат к 02-acceptance/03-fix
