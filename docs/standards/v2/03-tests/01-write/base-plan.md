# Базовый план: 03-tests-write

## Исполнители

Два агента работают **параллельно**:
- **Азирафаль** — happy path и contract тесты
- **Кроули** — adversarial тесты (edge cases, failure modes, граничные условия)

## 1. Сбор контекста

- Прочитай финальный acceptance: последний `02-acceptance/03-fix/report-*.md`, если нет — `02-acceptance/01-write/report-*.md`
- Прочитай spec для понимания архитектурного контекста
- Прочитай `PROJECT.md` — стек, используемые библиотеки, соглашения по тестированию

## 2. Структура файлов

Тесты **обязательно** разделяются по папкам:

```
tests/
  happy/       # Азирафаль: happy path, contract, ожидаемое поведение
  adversarial/ # Кроули: edge cases, failure modes, граничные условия
```

Каждый файл-тест — отдельный модуль. **Максимальный размер модуля: 100–150 строк** (без учёта комментариев). Если файл переваливает за порог — дроби по функциональным группам.

## 3. Принципы FIRST

Каждый тест должен быть:
- **Fast** — выполняется быстро; медленные зависимости — в интеграционных тестах, не юнит
- **Isolated** — не зависит от других тестов; не полагается на их порядок или результат
- **Repeatable** — детерминирован; нет `rand`, нет `time.Now()` без инъекции, нет сетевых вызовов без мока
- **Self-validating** — тест сам определяет pass/fail, не нужна ручная проверка
- **Timely** — тест пишется к acceptance, а не после того как код уже написан

## 4. Правила написания

### Поведение, не реализация

Тест проверяет **наблюдаемое поведение** — что возвращается, какой статус, какое состояние снаружи. Не проверяет внутренние детали — какой метод был вызван, сколько раз, в каком порядке. Это делает тест стабильным при рефакторинге.

### One test = one scenario

Один тест = один сценарий из acceptance. Имя теста = описание сценария + ожидаемый результат:
```
TestConnect_DatabaseUnavailable_ReturnsErrTimeout
TestConnect_ValidConfig_ReturnsConnection
```

### Own setup

Каждый тест сам подготавливает данные. Нет общего mutable state между тестами. Используй `t.Cleanup` / `defer` для очистки.

### Детерминизм

- Нет `time.Now()` — инжектируй `clock` интерфейс или принимай `now time.Time` параметром
- Нет `rand.Intn()` без seed — или инжектируй источник случайности, или фиксируй seed
- Нет сетевых вызовов — используй test doubles (mock, stub, fake) для внешних зависимостей

## 5. Азирафаль: happy path и contract тесты

- Покрывает каждый happy path сценарий из acceptance
- Проверяет контракт: правильные данные на правильный ввод
- Проверяет что поведение соответствует acceptance при нормальных условиях
- Файлы: `tests/happy/<feature>_test.<ext>`

## 6. Кроули: adversarial тесты

- Покрывает каждый failure mode из acceptance
- Граничные условия: nil, пустые значения, максимальные значения, переполнение
- Поведение при недоступности зависимостей
- Конкурентность: race conditions, deadlocks если применимо
- Идемпотентность если применимо
- Файлы: `tests/adversarial/<feature>_test.<ext>`

## 7. Проверка покрытия

После написания — сверь каждый сценарий из acceptance с тестами. Каждый сценарий должен быть покрыт хотя бы одним тестом.

Если сценарий невозможно покрыть тестом из-за неполноты acceptance — запиши `clarification`, откат к `02-acceptance/03-fix`.

## 7а. Red run (обязательно)

После написания тестов — запусти их **до реализации**.

`verify: тест падает без реализации (Red run)`

Если тест проходит без реализации — тест не проверяет то что должен. Это false positive: либо тест написан неверно, либо реализация уже существует и не соответствует ожиданиям. Разберись до продолжения.

## 8. Формирование отчёта

Report содержит:
- Список написанных тестов (happy и adversarial отдельно)
- Маппинг: тест → сценарий acceptance
- Непокрытые сценарии если есть
- Результаты первого прогона

## Движение вперёд и назад

**Done когда:** все сценарии acceptance покрыты, тесты проходят, report написан
**Следующий:** 03-tests/02-review
**Передаём:** тест-файлы + report-NNN.md

**Возврат назад:**
- Acceptance неполный или противоречит себе → `clarification`, откат к 02-acceptance/03-fix
