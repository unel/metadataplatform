# Базовый план: 03-tests-fix

## 1. Сбор контекста

- Прочитай `03-tests/02-review/report-*.md` — список замечаний с классификацией
- Прочитай тест-файлы с замечаниями
- Прочитай acceptance если нужно уточнить ожидаемое поведение

## 2. Определение исполнителя

- Замечания по `tests/adversarial/` → Кроули
- Замечания по `tests/happy/` → Азирафаль
- Если оба файла затронуты — работают параллельно каждый в своей зоне

## 3. Согласование плана

- Покажи план фикса пользователю: что именно меняется и почему
- Дождись подтверждения

## 4. Исправление

- Исправляй только то что указано в review
- Не переписывай тесты шире чем нужно для фикса
- Не добавляй новые тесты без явного указания (для этого есть отдельный test-write)
- Сохраняй структуру `tests/happy/` и `tests/adversarial/`
- Держи каждый модуль ≤ 150 строк

## 5. Невозможные фиксы

Если замечание нельзя исправить без изменения acceptance:
- Запиши `clarification` в report
- Объясни почему — что конкретно в acceptance мешает
- Не угадывай и не додумывай — откат к `02-acceptance/03-fix` через оркестратора

## 6. Формирование отчёта

- Список исправленных замечаний (ссылка → что сделано)
- Список неисправленных с объяснением
- После фикса — тесты должны проходить

## Движение вперёд и назад

**Done когда:** все исправимые замечания закрыты, тесты проходят, report написан
**Следующий:** 03-tests/02-review (повторное ревью)
**Передаём:** report-NNN.md

**Возврат назад:**
- Замечание требует изменения acceptance → `clarification`, откат к 02-acceptance/03-fix
