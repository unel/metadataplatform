# Базовый план: 03-tests-run

## Контекст

Это **TDD Red** шаг. Реализации ещё нет. Цель — убедиться что тесты корректно проверяют реализацию, а не пропускают её отсутствие.

Финальный прогон после написания кода — в `04-code/04-testing`.

## 1. Сбор контекста

- Проверь что review прошёл: `03-tests/02-review/report-*.md` со статусом `done`
- Прочитай `PROJECT.md` — команда для запуска тестов
- Убедись что реализации ещё нет (код фичи не написан)

## 2. Запуск

- Запусти тест-suite: `tests/happy/` и `tests/adversarial/`
- С детальным выводом (-v или аналог)

## 3. Интерпретация результатов

**Ожидаемый результат: все тесты падают.** Это норма, не проблема.

### Красный флаг: тест прошёл без реализации

Если тест **прошёл** при отсутствующей реализации — это серьёзная проблема:
- Тест всегда true (неправильный assert, пустое условие)
- Тест проверяет мок вместо реального кода
- Тест содержит логическую ошибку в условии

Такой тест → `failed`, откат к `03-tests/03-fix`.

### Тест упал по неожиданной причине

Если тест упал не из-за отсутствия реализации, а из-за compile error, паники в тест-хелпере, проблемы в acceptance — укажи причину в report, это может потребовать отката.

## 4. Формирование отчёта

- Итог: сколько упало / пропущено / (аномально) прошло
- Для каждого аномально прошедшего теста — объяснение почему это подозрительно
- Для неожиданных падений — причина

## Движение вперёд и назад

**Done когда:** все тесты красные по ожидаемой причине (нет реализации)
**Следующий:** 04-code/01-write
**Передаём:** report-NNN.md

**Возврат назад:**
- Тест прошёл без реализации → `failed`, откат к 03-tests/03-fix
- Acceptance неполный или противоречивый → `clarification`, откат к 02-acceptance/03-fix
