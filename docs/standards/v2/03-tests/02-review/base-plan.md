# Базовый план: 03-tests-review

## 1. Сбор контекста

- Прочитай тест-файлы в `tests/happy/` и `tests/adversarial/`
- Прочитай `03-tests/01-write/report-*.md` — что было написано и почему
- Прочитай финальный acceptance — для проверки покрытия

## 2. Проверка структуры

- Happy path тесты в `tests/happy/`, adversarial — в `tests/adversarial/`
- Каждый модуль ≤ 150 строк (без комментариев)
- Именование файлов и тестов читается как документация

## 3. Проверка FIRST

- **Fast**: юнит тесты не обращаются к сети, БД, файловой системе напрямую
- **Isolated**: нет общего mutable state между тестами; каждый тест самодостаточен
- **Repeatable**: нет `time.Now()` без инъекции, нет случайности без фиксированного seed
- **Self-validating**: тест сам определяет pass/fail без ручной интерпретации
- Нет скипнутых / закомментированных тестов без явного обоснования

## 4. Проверка поведения

- Тесты проверяют наблюдаемое поведение, не внутренние детали реализации
- Тесты не сломаются при рефакторинге внутренней реализации (пока контракт не меняется)

## 4а. Фальсифицируемость

Тест который не может провалиться при дефектной системе — не тест, а симулякр.

- **Таймаут-обёртки**: если тест содержит внешний таймаут (контекст с дедлайном, `time.After` на уровне теста, `select` с таймером) — внешний таймаут должен быть строго МЕНЬШЕ ожидаемого внутреннего. Если внешний таймаут больше или равен — тест завершится по обёртке независимо от поведения системы; deadlock и зависание пройдут незамеченными.
- **Тесты на негативное поведение**: задай вопрос: "что произойдёт если система ничего не делает — завис, deadlock, не ответила?" Если тест всё равно пройдёт — он нефальсифицируем. Для тестов на таймауты, graceful shutdown, отмену контекста это обязательная проверка.

## 5. Проверка покрытия

Сверь каждый сценарий acceptance с тестами:
- Каждый happy path покрыт хотя бы одним тестом Азирафаля
- Каждый failure mode покрыт тестом Кроули
- Граничные условия покрыты
- НФТ поддающиеся тестированию — покрыты

## 6. Классификация проблем

- **Пробел в покрытии** — сценарий из acceptance не покрыт тестом → test-fix
- **Нарушение FIRST** — тест не изолирован, не детерминирован → test-fix
- **Проверка реализации** — тест завязан на внутренние детали → test-fix
- **Неопределённость в acceptance** — сценарий нельзя покрыть тестом → `clarification`, откат к acceptance

## 7. Формирование отчёта

- Нет проблем → "Тесты чистые. Готово к code-write."
- Есть проблемы → список с классификацией, объяснением, ссылкой на тест/сценарий

## Движение вперёд и назад

**Done когда:** нет пробелов в покрытии, нет нарушений FIRST
**Следующий:** 04-code/01-write
**Передаём:** report-NNN.md

**Возврат назад:**
- пробелы в покрытии или нарушения FIRST → `failed`, переход к 03-tests/03-fix
- сценарий acceptance нельзя покрыть → `clarification`, откат к 02-acceptance/03-fix (через оркестратора)
