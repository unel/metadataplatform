# Базовый чек-лист: 04-code-review

## Контекст

- [ ] Код прочитан
- [ ] Spec и acceptance прочитаны для проверки соответствия

## Соответствие требованиям

- [ ] Код реализует всё из acceptance
- [ ] Нет лишней функциональности которой нет в spec (YAGNI / over-engineering)
- [ ] Все failure modes из acceptance обработаны

## Именование

- [ ] Функции/методы — глаголы: `uploadDocument()`, `findUserById()`
- [ ] Переменные/классы — существительные: `user`, `accountBalance`
- [ ] Булевые — с `is`/`has`/`can` префиксом: `isActive`, `hasPermission`
- [ ] Одно слово для одного понятия по всей базе — нет `fetch`/`get`/`retrieve` вперемешку
- [ ] Различие в именах раскрывает различие в смысле — нет `Data`/`Info` пар без смысла
- [ ] Нет магических чисел без именованных констант
- [ ] Имена произносимы и grep-абельны
- [ ] Нет венгерской нотации и `IInterface` префиксов
- [ ] Имена не требуют комментария чтобы понять зачем они

## Единый уровень абстракции (SLAP)

- [ ] Каждый метод — на одном уровне абстракции: нет смеси бизнес-оркестрации и деталей (SQL, HTTP, форматирование) в одном методе
- [ ] Нет комментариев `// Step 1 / Step 2` — верный признак что метод делает несколько вещей
- [ ] Функции в файле расположены сверху вниз от высокого уровня к низкому (Stepdown Rule)

## Структура

- [ ] Нет модулей > 150 строк (без комментариев)
- [ ] Каждый модуль — одна ответственность

## Читаемость

- [ ] Нет избыточной вложенности — применены early return / guard clauses
- [ ] Комментарии объясняют WHY, не WHAT
- [ ] Нет комментариев пересказывающих код
- [ ] Нет закомментированного кода
- [ ] Нет TODO без номера тикета

## Безопасность

- [ ] Нет SQL-инъекций — параметризованные запросы
- [ ] Нет command injection
- [ ] Нет секретов в коде
- [ ] Валидация пользовательского ввода на границах
- [ ] Нет избыточного логирования чувствительных данных

## Обработка ошибок

- [ ] Ошибки не проглатываются
- [ ] Нет паники для предсказуемых условий

## Классификация замечаний

- [ ] Каждое замечание классифицировано: critical / warning / Nit:
- [ ] critical и warning: указан файл:строка + объяснение
- [ ] Не исправлял код самостоятельно — только описал проблемы

## Движение дальше

- [ ] Нет critical и warning → report `done`, следующий 04-code/04-testing
- [ ] Есть critical или warning → report `failed`, следующий 04-code/03-fix
