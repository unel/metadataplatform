# Базовый план: 04-code-review

## 1. Сбор контекста

- Прочитай код реализации
- Прочитай `04-code/01-write/report-*.md`
- Прочитай spec и acceptance — для проверки соответствия требованиям

## 2. Соответствие требованиям

- Код реализует всё что описано в acceptance?
- Нет ли лишнего — функциональности которой нет в spec (over-engineering)?
- Все failure modes из acceptance обработаны?
- Если замечание вводит новое поле конфига или меняет поведение — spec.md и acceptance.md должны быть обновлены в том же code-fix. Ревью не закрывается без этого обновления.

## 3. Именование

Именование — первый сигнал о качестве кода. Плохое имя — это скрытый баг в понимании.

### Intention-Revealing Names (Martin, Clean Code)

> "A name should tell you why it exists, what it does, and how it is used. If a name requires a comment, then the name does not reveal its intent."

Если имя требует комментария — имя неправильное.

### Конкретные правила

- **Глагол для функций/методов**: `uploadDocument()`, `calculateTotal()`, `findUserById()`
- **Существительное для переменных и классов**: `user`, `accountBalance`, `OrderRepository`
- **Булевые** — с `is`/`has`/`can` префиксом: `isActive`, `hasPermission`, `canRetry`
- **One Word Per Concept**: одно понятие — одно слово по всей базе. Либо `fetch`, либо `get`, но не оба вперемешку
- **Meaningful Distinctions**: различие в имени раскрывает различие в смысле. `ProductInfo` vs `ProductData` — не различие
- **Searchable**: нет магических чисел без именованных констант — `7` vs `MAX_RETRIES`
- **Pronounceable**: имя должно произноситься вслух — если нельзя произнести, нельзя обсудить
- **Без encoding**: венгерская нотация (`strName`, `iCount`) и `IInterface` префиксы — мусор
- **Классы**: не `Manager`, `Processor`, `Data`, `Info` — эти слова ничего не говорят

## 4. Единый уровень абстракции (SLAP)

> "Each method should be written in terms of a single level of abstraction." — Robert C. Martin, Clean Code

Метод не смешивает *что* делается (бизнес-оркестрация) с *как* это делается (SQL, HTTP, форматирование строк).

**Признаки нарушения:**
- Комментарии `// Step 1: ...`, `// Step 2: ...` — верный признак что метод делает несколько вещей на разных уровнях
- Рядом с высокоуровневым вызовом `confirmOrder(order)` стоит низкоуровневый `db.execute("UPDATE ...")`
- Один метод и парсит ответ HTTP, и обновляет состояние, и форматирует вывод

**Stepdown Rule**: функции в файле располагаются сверху вниз в порядке убывания уровня абстракции — читая файл, спускаешься на один уровень за раз.

## 5. Структура и размер

- Нет модулей > 150 строк (без комментариев) — если есть, модуль делает слишком много
- Каждый модуль — одна ответственность (SRP)
- Структура директорий соответствует конвенциям проекта

## 6. Читаемость

- Нет избыточной вложенности — применены early return / guard clauses
- Комментарии объясняют **WHY**, не WHAT: хорошо написанный код самодокументирован
- Нет комментариев пересказывающих код (`// increment i`)
- Нет закомментированного кода — удалять, для этого есть git
- TODO без номера тикета — не принимается

## 7. SOLID и архитектура

- SRP: каждый модуль — одна причина меняться
- Нет зависимостей от конкретных реализаций там где нужна абстракция (DIP)
- Интерфейсы не требуют реализовывать лишнее (ISP)

## 8. Безопасность

- Нет SQL-инъекций — параметризованные запросы
- Нет command injection
- Нет секретов в коде
- Валидация на границах системы (пользовательский ввод, внешние API)
- Нет избыточного логирования чувствительных данных

## 9. Обработка ошибок

- Ошибки не проглатываются
- Сообщения об ошибках информативны для вызывающей стороны, не раскрывают внутреннее
- Нет паники для предсказуемых условий

## 10. Go-сервисы (обязательно в каждом раунде)

Применяется к каждому раунду ревью Go-сервиса — не только к первому.

- Прочитаны ВСЕ файлы пакета, включая `main.go` — даже если они не менялись в этом раунде
- **Signal handling**: как регистрируются сигналы, идемпотентна ли операция `Stop`, что происходит при двойном сигнале (double-signal)
- **Инициализация**: порядок инициализации компонентов, обработка ошибок на каждом шаге, cleanup при неудаче одного из шагов
- **Shutdown path**: если в этом раунде менялась горутинная структура — пройти shutdown path целиком как отдельный шаг; убедиться что нет deadlock и goroutine leak

## 11. Классификация замечаний

- **critical** — нарушение безопасности, потеря данных, несоответствие acceptance; блокирует
- **warning** — нарушение SLAP, плохое именование, модуль > 150 строк, плохая обработка ошибок; надо исправить
- **Nit:** — стиль, мелкие улучшения; не блокирует, но желательно (префикс `Nit:` в описании)

## 12. Формирование отчёта

- Нет critical/warning → "Код чистый. Готово к финальному прогону тестов."
- Есть замечания → список с классификацией, файл:строка, объяснение

## Движение вперёд и назад

**Done когда:** нет замечаний critical и warning
**Следующий:** 04-code/04-testing (финальный прогон)
**Передаём:** report-NNN.md

**Возврат назад:**
- Есть critical или warning → `failed`, переход к 04-code/03-fix
- Код не соответствует acceptance → передай оркестратору на уточнение
