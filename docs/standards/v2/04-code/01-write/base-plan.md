# Базовый план: 04-code-write

## 1. Сбор контекста

- Прочитай тесты: `tests/happy/` и `tests/adversarial/` — они определяют контракт реализации
- Прочитай финальный acceptance
- Прочитай spec для понимания архитектурного контекста
- Прочитай `PROJECT.md` — стек, соглашения по структуре кода

## 2. Цель: заставить тесты пройти

Пиши **минимально необходимый код** чтобы все тесты прошли. Не больше:
- Нет функциональности которую не проверяет ни один тест
- Нет "на будущее" — YAGNI
- Нет абстракций которые не нужны прямо сейчас

Тесты — это спецификация. Если тест требует поведения X, реализуй X. Если тест ничего не говорит про Y — не реализуй Y.

## 3. Структура и размер модулей

**Максимальный размер модуля: 100–150 строк** (без учёта комментариев). Превышение — сигнал что модуль делает слишком много. Дроби по ответственностям.

Каждый модуль — одна ответственность (SRP). Называй по тому что делает, не по тому что содержит.

## 4. Принципы написания

### Читаемость

- Имена переменных, функций и типов — самодокументирующие; комментарии только когда WHY неочевидно
- Нет многоуровневой вложенности — early return, guard clauses
- Нет магических чисел и строк — именованные константы

### SOLID

- **SRP**: один модуль — одна причина меняться
- **OCP**: расширяй через добавление, не через изменение существующего
- **LSP**: подтипы не нарушают контракт базового типа
- **ISP**: интерфейсы узкие — не вынуждай реализовывать то что не нужно
- **DIP**: зависи от абстракций, не от конкретных реализаций

### Безопасность (OWASP)

- Нет SQL-инъекций — параметризованные запросы
- Нет command injection — не собирай команды из пользовательского ввода
- Нет хранения секретов в коде
- Валидация на границах системы (пользовательский ввод, внешние API)
- Нет избыточного логирования чувствительных данных

### Обработка ошибок

- Ошибки обрабатываются явно, не проглатываются
- Сообщения об ошибках — информативны для вызывающей стороны, не раскрывают внутренние детали
- Нет паники / unhandled exception для предсказуемых условий

## 5. Запуск тестов после написания

После реализации — запусти тесты. Ожидаемый результат: все зелёные.

Если тест упал:
- Из-за ошибки в коде → исправь код
- Из-за неясного требования в acceptance → `clarification`, эскалируй

Не меняй тесты чтобы они прошли — меняй код.

## 6. Формирование отчёта

- Список созданных / изменённых файлов
- Результаты финального прогона тестов

## Движение вперёд и назад

**Done когда:** все тесты зелёные, report написан
**Следующий:** 04-code/02-review
**Передаём:** код + report-NNN.md

**Возврат назад:**
- Тест требует поведения не описанного в acceptance → `clarification`, откат к 03-tests/03-fix или 02-acceptance/03-fix
