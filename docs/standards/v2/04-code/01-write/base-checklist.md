# Базовый чек-лист: 04-code-write

## Контекст

- [ ] Тесты прочитаны — контракт реализации понят
- [ ] Acceptance прочитан
- [ ] `PROJECT.md` прочитан — стек, соглашения

## Минимальность и трассировка

- [ ] Код реализует только то что требуют тесты — нет функциональности "на будущее"
- [ ] Нет абстракций без обоснования в текущих требованиях (YAGNI)
- [ ] Каждый блок кода трассируется к конкретному пункту acceptance — нет изменений без трассировки
- [ ] Нет изменений в коде вне области задачи (соседние функции/файлы не связанные с AC)
- [ ] Все допущения при неясности спеки — сформулированы явно и подтверждены через `/ask` до реализации

## Структура модулей

- [ ] Каждый модуль ≤ 150 строк (без комментариев)
- [ ] Каждый модуль — одна ответственность (SRP)

## Читаемость

- [ ] Имена самодокументирующие — не нужен комментарий чтобы понять что делает
- [ ] Нет магических чисел и строк — используются именованные константы
- [ ] Нет глубокой вложенности — применены early return / guard clauses

## Безопасность

- [ ] Нет SQL-инъекций — параметризованные запросы
- [ ] Нет command injection
- [ ] Нет секретов в коде
- [ ] Валидация пользовательского ввода на границах
- [ ] Нет логирования чувствительных данных

## Обработка ошибок

- [ ] Ошибки обрабатываются явно — не проглатываются
- [ ] Нет паники / unhandled exception для предсказуемых условий

## Тесты

- [ ] Все тесты (`tests/happy/` и `tests/adversarial/`) зелёные — `verify: Green run`
- [ ] Тесты не изменялись чтобы пройти — менялся только код
- [ ] После любого рефакторинга структуры — `verify: поведение не изменилось, тесты зелёные`

## Движение дальше

- [ ] Все тесты зелёные → report написан, следующий 04-code/02-review
- [ ] Тест требует поведения вне acceptance → `clarification`, эскалировано
