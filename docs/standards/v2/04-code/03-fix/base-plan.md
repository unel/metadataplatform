# Базовый план: 04-code-fix

## 1. Сбор контекста

- Прочитай `04-code/02-review/report-*.md` — список замечаний с классификацией
- Прочитай затронутые файлы кода

## 2. Согласование плана

- Покажи план фикса пользователю: что именно меняется и почему
- Дождись подтверждения

## 3. Исправление

Исправляй только то что указано в review. Не рефакторь то что не затронуто замечаниями.

Приоритет: сначала `critical`, потом `warning`, потом `note`.

После каждого исправления — держи тесты зелёными. Если фикс ломает тест — это сигнал что фикс неверен или тест неполный.

Держи каждый модуль ≤ 150 строк. Если фикс требует разбиения модуля — разбей.

## 4. Невозможные фиксы

Если замечание нельзя исправить без изменения acceptance или spec:
- Запиши `clarification` в report
- Объясни: что конкретно в acceptance/spec мешает
- Откат через оркестратора к `02-acceptance/03-fix`

## 5. Проверка после фикса

Запусти тесты. Ожидаемый результат: все зелёные (как были до фикса).

## 6. Формирование отчёта

- Список исправленных замечаний (ссылка → что сделано)
- Список неисправленных с объяснением

## Движение вперёд и назад

**Done когда:** все исправимые замечания закрыты, тесты зелёные, report написан
**Следующий:** 04-code/02-review (повторное ревью)
**Передаём:** report-NNN.md

**Возврат назад:**
- Замечание требует изменения acceptance → `clarification`, откат к 02-acceptance/03-fix
