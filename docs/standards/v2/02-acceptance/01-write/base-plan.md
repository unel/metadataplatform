# Базовый план: 02-acceptance-write

## 1. Сбор контекста

- Прочитай актуальную spec: последний `01-spec/03-fix/report-*.md`, если нет — `01-spec/01-write/report-*.md`
- Прочитай `PROJECT.md` — архитектурные ограничения влияющие на acceptance
- Прочитай `PLAN.md` если нужно понять приоритеты фичи

## 2. Составление сценариев

Сценарии в формате Given/When/Then для:
- каждого happy path из ФТ
- каждого failure mode из ФТ
- каждого граничного условия
- НФТ которые можно проверить тестом (производительность, параллелизм, идемпотентность)

### Строгие правила Given/When/Then

**Given** — только контекст (предусловие):
- Описывает состояние системы и данных перед действием
- НЕ содержит действий — только "система находится в состоянии X"
- Пример хорошо: `Given соединение с БД установлено, таблица содержит 5 записей`
- Пример плохо: `Given пользователь открыл соединение` — это действие

**When** — ровно одно действие:
- Одно действие вызывающей стороны, без "And" в When
- Пример хорошо: `When вызван Connect(config)`
- Пример плохо: `When вызван Connect(config) And переданы credentials` — два действия

**Then** — наблюдаемый результат с точки зрения вызывающей стороны:
- Что возвращается, какое состояние снаружи — НЕ что происходит внутри
- Пример хорошо: `Then возвращается ошибка ErrTimeout`
- Пример плохо: `Then система делает retry к DB` — внутренняя деталь

### User-centric формулировки

Acceptance описывает что вызывающая сторона наблюдает, не как система это делает внутри:
- Хорошо: "возвращает HTTP 503 с телом `{error: "unavailable"}`"
- Плохо: "вызывает метод Fallback у зависимости X"

### Измеримость

Результаты должны быть проверяемы без интерпретации:
- Хорошо: "завершается за ≤200мс при p99"
- Плохо: "работает быстро"
- Хорошо: "возвращает ошибку `ErrConnectionRefused`"
- Плохо: "показывает понятное сообщение об ошибке"

### Формула сильного критерия

Слабый критерий — неверифицируем:

> "система корректно обрабатывает unix socket"

Сильный критерий называет: тест, действие, наблюдаемый результат, временной порог если применимо:

> "TestConnectUnixSocket запускается, отправляет пакет, получает ответ без ошибок — в рамках 100мс"

Проверяй каждый Then-блок вопросом: "Можно ли написать тест который однозначно проходит или падает по этому критерию — без интерпретации?" Если нет — критерий слабый.

### Независимость сценариев

Каждый сценарий самодостаточен:
- Given описывает всё необходимое состояние — не полагается на предыдущий сценарий
- Сценарии можно выполнять в любом порядке

## 3. Проверка на пробелы

При составлении сценариев — обращай внимание: появился ли новый failure mode которого нет в spec?

Если да — **не угадывай**, записывай `clarification` и сообщай пользователю: нужен откат к `01-spec/03-fix`.

## 4. Проверка anti-patterns

Перед согласованием убедись, что нет:
- Расплывчатых Then: "работает корректно", "обрабатывает правильно", "успешно"
- Реализационных Then: "вызывает метод X", "делает запрос к БД", "пишет в лог"
- Нескольких действий в When (And в When)
- Действий в Given (Given пользователь нажал...)
- Сценариев зависящих от порядка или результатов других сценариев

## 5. Согласование

- Покажи черновик acceptance пользователю
- Жди подтверждения или правок

## 6. Формирование отчёта

После подтверждения — запиши отчёт содержащий полный acceptance.

## Формат сценариев

```markdown
## Сценарий: <название>

**Given** <начальное состояние системы и данных>
**When** <одно действие вызывающей стороны>
**Then** <наблюдаемый результат — конкретный, измеримый>
```
