## Строгость Given/When/Then

- [ ] **Given** — только контекст (предусловие), не содержит действий
- [ ] **When** — ровно одно действие вызывающей стороны; нет "And" в When
- [ ] **Then** — наблюдаемый результат с точки зрения вызывающей стороны; нет внутренней реализации

## Качество сценариев

- [ ] Формулировки user-centric: что возвращается / что наблюдается, не как система это делает внутри
- [ ] Результаты измеримы: конкретные коды ошибок, числовые пороги, не "быстро" / "понятно" / "корректно"
- [ ] Входные данные конкретны — не "valid input" / "некорректные данные"
- [ ] Один сценарий = одна проверка
- [ ] Нет сценариев проверяющих очевидное (1=1, "возвращает то что передали")
- [ ] Сценарии независимы — каждый самодостаточен, не зависит от порядка или результата других

## Anti-patterns (ничего из этого нет)

- [ ] Нет расплывчатых Then: "работает корректно", "обрабатывает правильно", "успешно"
- [ ] Нет реализационных Then: "вызывает метод X", "делает запрос к БД", "пишет в лог"
- [ ] Нет нескольких действий в When
- [ ] Нет действий в Given

## Сетевые протоколы (если фича работает с сетью)

- [ ] Full-duplex или half-duplex? Кто инициирует поток данных?
- [ ] Что происходит если одна сторона перестаёт читать?
- [ ] Есть ли backpressure? Как проявляется при заполнении буфера?
- [ ] Что при объёме данных > kernel socket buffer (~8-208 КБ для Unix socket)?
- [ ] Кто закрывает соединение и в каком порядке?
- [ ] Как воспроизвести каждый сценарий в коде теста — инструменты, mock-точки, границы?
  _(если невозможно автоматически → пометить "manual only" или "требует mock на уровне X")_
