# Базовый чек-лист: 02-acceptance-review

## Контекст

- [ ] Spec прочитана (актуальная версия)
- [ ] Acceptance прочитан из `02-acceptance/01-write/report-*.md`

## Покрытие

- [ ] Каждый happy path из ФТ имеет сценарий
- [ ] Каждый failure mode из ФТ покрыт
- [ ] Граничные условия покрыты
- [ ] Поведение при недоступности каждой зависимости покрыто
- [ ] Проверяемые НФТ покрыты

## Качество Given/When/Then

- [ ] **Given** — только контекст, не содержит действий
- [ ] **When** — ровно одно действие; нет "And" в When
- [ ] **Then** — наблюдаемый результат вызывающей стороны; нет внутренних деталей реализации

## Качество формулировок

- [ ] User-centric: что возвращается/наблюдается, не как система делает это внутри
- [ ] Измеримо: конкретные коды ошибок, числовые пороги — не "быстро" / "корректно" / "понятно"
- [ ] Входные данные конкретны — нет "valid input"
- [ ] Один сценарий = одна проверка
- [ ] Сценарии независимы — не зависят от порядка или результатов других
- [ ] Нет тривиальных сценариев (1=1, "возвращает то что передали")

## Adversarial покрытие

- [ ] Что при объёме данных x1000 от тестируемого? Есть ли сценарий с kernel buffer overflow?
- [ ] Каждый сценарий с сетевым взаимодействием — прошёл чек-лист из acceptance-write (full-duplex, backpressure, воспроизводимость в тесте)?
- [ ] Сценарии которые нельзя воспроизвести автоматически — явно помечены "manual only"?

## Классификация найденных проблем

- [ ] Каждая проблема классифицирована: пробел в acceptance / неопределённость в spec / противоречие в spec / проблема качества
- [ ] Проблемы требующие отката к spec — отмечены явно
- [ ] Не исправлял acceptance и spec самостоятельно — только описал проблемы

## Отчёт

- [ ] Если нет проблем — явно написано "Acceptance чистый"
- [ ] Если есть — каждая проблема с объяснением и ссылкой на сценарий
