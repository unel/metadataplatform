# Базовый план: 05-docs-review

## 1. Сбор контекста

- Прочитай документационные файлы из `05-docs/01-write/report-*.md`
- Прочитай код реализации — для проверки точности утверждений
- Прочитай spec и acceptance — для проверки полноты

## 2. Проверка точности

- Примеры в документации рабочие и соответствуют реальному поведению кода?
- Описанные коды ошибок и форматы ответов соответствуют реализации?
- Нет устаревших утверждений которые противоречат коду?

## 3. Проверка типов (Diataxis)

- How-to не объясняет "почему так" — только шаги к результату?
- Reference нейтральный — нет советов и мнений, только факты?
- Explanation / ADR содержит поле Context с реальными constraints?
- Типы не смешаны в одном документе?

## 4. Проверка полноты

- Описано что делает фича (what)?
- Описано как использовать — с примерами (how)?
- Описано поведение при ошибках (failure modes из acceptance)?
- Если есть нетривиальные технические решения — они задокументированы?
- API endpoint'ы (если есть) — задокументированы?

## 4. Проверка читаемости

- Написано для читателя а не автора — отвечает на реальные вопросы?
- Есть конкретные примеры, не абстрактные описания?
- Нет очевидного (документирования того что понятно из имён)?
- Описывает текущее состояние, не историю?

## 5. Классификация замечаний

- **critical** — неточная/вводящая в заблуждение информация, пример который не работает
- **warning** — пропущено важное поведение, нет примеров, устаревшее описание
- **Nit:** — стиль, формулировки, структура

## 6. Формирование отчёта

- Нет critical/warning → "Документация чистая."
- Есть замечания → список с классификацией и объяснением

## Движение вперёд и назад

**Done когда:** нет замечаний critical и warning
**Следующий:** 06-retro/01-recall
**Передаём:** report-NNN.md

**Возврат назад:**
- Есть critical или warning → `failed`, переход к 05-docs/03-fix
