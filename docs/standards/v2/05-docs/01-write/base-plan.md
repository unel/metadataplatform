# Базовый план: 05-docs-write

## 1. Сбор контекста

- Прочитай spec и acceptance — что было сделано и зачем
- Прочитай код реализации — для технических деталей
- Проверь `PROJECT.md` — соглашения по документации в проекте
- Проверь существующую документацию — что уже есть, что нужно обновить

## 2. Четыре типа документации (Diataxis)

Каждый тип отвечает на разный вопрос и пишется для разной аудитории. **Держи типы раздельными** — смешивание это главная ошибка в документации.

### How-to guide — "как сделать X"

Для разработчика который знает что хочет, но не знает как. Шаги к конкретному результату. Не объясняет почему — только направляет.

Пиши для каждой нетривиальной операции с фичей: как подключить, как настроить, как обработать ошибку.

### Reference — точная нейтральная информация

Для разработчика который ищет конкретный факт. API endpoint'ы, параметры, коды ошибок, форматы. Никаких советов, никаких мнений — только точные данные.

### Explanation — "почему так"

Единственный тип где уместно иметь мнение. Трейдоффы, архитектурные решения, контекст. Пиши когда есть нетривиальные решения которые могут удивить следующего разработчика.

В виде ADR (Architecture Decision Record) — если решение важное и не очевидное:
- **Context**: что за проблема, какие были constraints — **это самое важное поле**, через год никто не вспомнит
- **Decision**: что решили
- **Consequences**: все последствия — позитивные, негативные, нейтральные
- **Considered options**: что рассматривалось и почему отброшено

ADR живёт в `docs/adr/`. Append-only: не редактируется. Если решение изменилось — новый ADR, старый получает статус "superseded by #N".

### Tutorial

Нужен только если фича настолько нова что требует ввода "с нуля". Для большинства фич не нужен — достаточно how-to.

## 3. README — структура

Если фича требует README:

1. **Что это и зачем** — первые три предложения. Читатель за 10 секунд должен понять стоит ли читать дальше
2. **Usage example с кодом** — до installation, не после. Реальный пример, не описание
3. **Installation / prerequisites** — если применимо
4. **Поведение при ошибках** — какие ошибки возможны, что с ними делать

Антипаттерн: начинать с build/dev инструкций — это для разработчиков проекта, не пользователей фичи.

## 4. Принципы

**Для читателя, не для автора** — документация отвечает на вопросы читателя. Знай аудиторию: junior-разработчик и DevOps читают разные документы.

**Incorrect docs хуже чем no docs** — устаревшая документация активно вредит. Если меняется поведение — обновляй, не добавляй рядом.

**Docs as Code** — документация живёт в репозитории рядом с кодом, проходит review. Не в Confluence, не в Google Docs.

**Progressive disclosure** — обзор сначала, детали по ссылке. Не сваливать всё в один документ.

**Конкретно** — с примерами кода, конкретными значениями, кодами ошибок. "Возвращает 404 если entity не найдена" лучше чем "обрабатывает ошибки корректно".

**Без очевидного** — не документируй то что понятно из имён и типов.

## 5. Что пишем для каждой фичи

Минимальный набор:
- How-to guide: как использовать фичу (если нетривиально)
- Reference: API / интерфейс / коды ошибок

По необходимости:
- Explanation / ADR: если есть архитектурные решения которые удивят следующего разработчика
- README обновление: если меняется публичный интерфейс

Не пишем если нечего добавить — плохая документация хуже отсутствующей.
