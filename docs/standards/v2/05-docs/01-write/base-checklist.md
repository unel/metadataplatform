# Базовый чек-лист: 05-docs-write

## Контекст

- [ ] Spec и acceptance прочитаны
- [ ] Код реализации прочитан
- [ ] Существующая документация проверена — что уже есть, что нужно обновить

## Типы документации (Diataxis)

- [ ] **How-to guide**: написан для нетривиальных операций с фичей
- [ ] **Reference**: API / параметры / коды ошибок — точно, нейтрально, без советов
- [ ] **Explanation / ADR**: написан если есть архитектурные решения которые удивят следующего разработчика
  - Context (constraints, давление) — самое важное поле
  - Decision + Consequences + Considered options
  - Файл в `docs/adr/` если применимо
- [ ] Типы не смешаны — how-to не объясняет почему, reference не даёт советы

## README (если применимо)

- [ ] Первые три предложения: что это, зачем, как работает
- [ ] Usage example с кодом — до installation
- [ ] Поведение при ошибках описано

## Качество

- [ ] Для читателя, не для автора — отвечает на реальные вопросы аудитории
- [ ] Конкретно — примеры кода, конкретные значения, коды ошибок
- [ ] Без очевидного — не документирует то что понятно из имён
- [ ] Описывает текущее состояние, не историю
- [ ] Существующая документация обновлена если поведение изменилось (не добавлена рядом)

## Отчёт

- [ ] Список созданных / обновлённых файлов с описанием что добавлено
