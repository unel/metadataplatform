# Стандарт процесса v2

## Принцип

Каждый **процесс** — атомарный шаг workflow с полным набором мета-артефактов. Процесс самодостаточен: всё необходимое для его выполнения, отслеживания и ретроспекции хранится вместе с ним.

## Структура директории процесса

```
tasks/<feature>/processes/<group>/<process>/
  status-log.md          — append-only хронология статусов; оркестратор читает последнюю запись
  README.md              — снимок стандарта процесса с метаданными фичи
  checklist.md           — сгенерированный чек-лист прогона (базовый + расширения)
  notes-<агент>.md       — заметки участника (дополняется, не перезаписывается)
  complaints-<агент>.md  — жалобы участника (дополняется, не перезаписывается)
  report-NNN.md          — иммутабельный артефакт прогона N
```

## README.md процесса

При инициализации — копируется из `docs/standards/v2/<group>/<process>/README.md` в директорию процесса фичи. Добавляются метаданные:

```markdown
---
feature: <путь к фиче>
generated: <datetime>
source: docs/standards/v2/<group>/<process>/README.md
---
```

## Чек-листы: базовый + расширения

Каждый процесс имеет базовый чек-лист. Первый пункт всегда:

> **Проанализируй задействованные системы и подсистемы. Составь список дополнительных чек-листов которые нужно применить к этому прогону.**
>
> Примеры расширений: network-stack, unix-socket, filesystem, concurrency/goroutines, signal-handling, config-parsing, database...

Агент принимает это решение самостоятельно по spec и acceptance.

Базовые чек-листы: `docs/standards/v2/<group>/<process>/base-checklist.md`
Расширения: `docs/standards/checklists/extensions/<system>.md`

Сгенерированный `checklist.md` — неизменный снимок с версиями источников:

```markdown
---
generated: 2026-04-26T18:00
base: code-review v1.2.0
extensions:
  - concurrency v1.0.1
  - unix-socket v1.1.0
---
```

## Статус-лог (`status-log.md`)

Append-only хронологический журнал. Никогда не перезаписывается — только дополняется. **Актуальный статус — последняя запись в файле.**

### Состояния

| Статус | Когда |
|---|---|
| `pending` | ожидает старта |
| `in-progress` | выполняется |
| `done` | завершён успешно |
| `failed` | завершён с ошибкой, нужен фикс |
| `stale` | был `done`, но вышестоящий артефакт изменился |
| `clarification` | входящие артефакты неясны или противоречивы; процесс приостановлен |

Правила:
- Каждый переход статуса — новая запись с datetime
- Одна строка комментария обязательна при `failed`, `stale`, `clarification`
- Ссылка на report при завершении прогона (`→ report-NNN.md`)

### Формат

```markdown
# 2026-04-25 18:00 — pending
Ожидает завершения spec-review.

# 2026-04-25 18:15 — in-progress
Гримм начал ревью. Расширения: concurrency-checklist, unix-socket-checklist.

# 2026-04-25 19:00 — failed
3 критичных замечания. → report-001.md

# 2026-04-25 19:30 — in-progress
Ада выполняет code-fix-2.

# 2026-04-25 20:00 — stale
code-fix-2 изменил горутинную структуру — нужен повторный проход shutdown path.

# 2026-04-25 21:00 — done
Все замечания закрыты. → report-003.md
```

## Формат репорта (`report-NNN.md`)

Пишется при каждом завершении прогона (`done`, `failed`, `clarification`). Иммутабелен после записи. Нумерация сквозная в рамках процесса.

```markdown
---
purpose: <одна строка — что произведено и кто читает>
process: <group/process>
run: <N>
date: <datetime>
created: <datetime>
status: done | failed | clarification
agent: <агент>
checklist: все пункты закрыты | открытые: <список>
see-also:
  - <путь/к/report-001.md>  # если предыдущий прогон остаётся актуальным контекстом
---

## Что сделано
- ...

## Результат
<что нашли / что изменили / что неясно>

## Артефакты
- произведены: [список файлов]
- входящие использованы: [список]

## Решения принятые в процессе
- ...

## Открытые вопросы
- ...
```

### `see-also`

Добавляется только если предыдущие репорты содержат актуальную информацию, не перекрытую текущим. Читающий обязан прочитать все `see-also` до начала работы.

## Notes и complaints: писать по ходу работы

`notes-<агент>.md` и `complaints-<агент>.md` — живые файлы. Агент дополняет их **во время выполнения процесса**, пока наблюдение свежее. Recall на ретро только добавляет ретроспективный взгляд — он не заменяет то что уже написано.

### `notes-<агент>.md` — конструктивные наблюдения

Одна строка = одно наблюдение с тегом:

```
[rework]   пришлось переделывать — что и почему
[friction] что замедляло работу
[miss]     что забыли или пропустили, обнаружилось позже
[propose]  предложение по процессу или инструментам
[doc]      нюанс который стоит задокументировать
[whatever] не вписывается — разберёмся на ретро
```

Когда писать: сразу как заметил — после фикса, после неожиданного решения, после того как что-то сломалось.

### `complaints-<агент>.md` — сырые жалобы

Свободная форма, без тегов, без фильтра. Что раздражало, что казалось бессмысленным, что потребовало лишних усилий. Писать прямо сейчас, не копить до ретро.

### Правила

- Оба файла — append-only, не перезаписываются
- Хранятся в директории процесса: `tasks/<feature>/processes/<group>/<process>/`
- На ретро (`06-retro/01-recall`) агент читает их и **дополняет** ретроспективным взглядом

## Поля `purpose`, `created`, `updated`

Каждый MD-файл в `tasks/` содержит в YAML-метаданных:
- `purpose` — одна строка, зачем файл существует и кто его читает
- `created` — проставляется один раз при создании
- `updated` — обновляется при каждом изменении (кроме репортов — они иммутабельны)

## Правила движения: вперёд и назад

Каждый процесс явно определяет в `base-plan.md` два блока:

**Движение вперёд:**
- Критерии `done`
- Следующий процесс
- Что передаётся следующему

**Возврат назад:**
- Триггеры отката
- Куда откатываемся
- Что делать перед откатом: записать причину в `status-log.md`, передать оркестратору

Пример для `02-acceptance/02-review`:
```
## Движение вперёд
done когда: нет замечаний "пробел в acceptance" или "проблема в spec"
следующий: 03-tests/01-write
передаём: report-NNN.md

## Возврат назад
если "пробел в acceptance" → 02-acceptance/03-fix
если "неопределённость в spec" → 01-spec/03-fix (через оркестратора)
если "противоречие в spec" → 01-spec/03-fix (через оркестратора)
```

## `clarification`: обнаружение проблемы в upstream

Если агент обнаруживает что входящие артефакты неполны или противоречивы — ставит `clarification` **только своему процессу** (downstream не трогает) и передаёт управление оркестратору с описанием проблемы.

**Оркестратор** решает куда перекинуть активность: к acceptance, к spec, или ещё выше.

**Агент на целевом upstream-шаге:**
1. Ставит своему процессу `in-progress`, пишет почему вернулись
2. Ставит `stale` последующим шагам **своей группы** (если были `done`/`failed`)
3. Рапортует оркестратору — тот каскадирует `stale` на все последующие группы

Пример:

```
04-code/02-review → clarification
  "CR-5 требует MaxConnections но spec не описывает поведение при лимите"
  ↓ оркестратор решает: идём к spec
01-spec/01-write → in-progress
  "Возврат из code-review: уточняем поведение при MaxConnections"
01-spec/02-review → stale  (агент ставит сам, в своей группе)
01-spec/03-fix → stale     (агент ставит сам, в своей группе)
  ↓ агент рапортует оркестратору: "изменил spec, каскадируй stale дальше"
02-acceptance/* → stale    (оркестратор)
03-tests/* → stale         (оркестратор)
04-code/* → stale          (оркестратор, включая тот process что стоял в clarification)
  ↓ оркестратор возобновляет pipeline с 01-spec/02-review
```

После прохождения всей цепочки управление естественно дойдёт обратно до `04-code/02-review`.

## `stale`: кто ставит, когда и как далеко

### Правило параллелизма

В каждый момент времени активен **только один процесс одной группы**. Это исключает конфликты при записи в `status-log.md` и делает каскадные изменения предсказуемыми.

### Кто ставит `stale`

**Исполнитель** (агент, работающий внутри группы) — ставит `stale` последующим шагам **в пределах своей группы**, если в ходе работы изменил артефакты предыдущих шагов той же группы. Затем рапортует оркестратору для межгруппового каскада.

```
Ада дописывает 01-spec/01-write (возврат из clarification) →
  ставит stale на 01-spec/02-review и 01-spec/03-fix (своя группа)
  рапортует оркестратору: "spec изменён, нужен межгрупповой stale"
```

**Оркестратор** — ставит `stale` при **межгрупповом каскаде**: когда исполнитель рапортует об изменении, оркестратор проверяет статусы последующих групп и ставит `stale` всем `done`/`failed` шагам далее по цепочке.

### Триггеры

| Событие | Кто ставит `stale` | Куда |
|---|---|---|
| Fix завершён внутри группы | Оркестратор | Все последующие группы (если были `done`/`failed`) |
| Агент изменил артефакт предыдущего шага своей группы | Агент | Последующие шаги своей группы; затем рапортует оркестратору |
| Пользователь вручную отредактировал артефакт | Оркестратор (по команде пользователя) | Все зависимые процессы начиная от изменённого |

### Каскадный сброс вперёд

Запись в `status-log.md` затронутых процессов:

```markdown
# 2026-04-27 14:00 — stale
Upstream артефакт изменён: spec-fix-2 уточнил поведение MaxConnections. Требует повторного прохода.
```

При следующем обходе pipeline `stale` трактуется как `pending` — шаг выполняется заново.

### Цепочка зависимостей

```
01-spec/01-write
  → 01-spec/02-review
  → 02-acceptance/01-write → 02-acceptance/02-review
  → 03-tests/01-write → 03-tests/02-review → 03-tests/04-run
  → 04-code/01-write → 04-code/02-review → 04-code/04-testing
  → 05-docs/01-write → 05-docs/02-review
```
