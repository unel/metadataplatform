# Базовый чек-лист: 01-spec-write

## Контекст

- [ ] Прочитан `TASK.md`
- [ ] Прочитан `PROJECT.md`
- [ ] Прочитан `PLAN.md`
- [ ] `00-research/01-interview/report-*.md` прочитан (если существует)
- [ ] `00-research/02-web/report-*.md` прочитан (если существует)
- [ ] Смежные spec прочитаны (если есть влияющие)

## Функциональные требования

- [ ] Happy path описан конкретно — не "система обрабатывает X", а что именно происходит
- [ ] Ограничения MVP явно перечислены
- [ ] Все входные данные описаны с примерами
- [ ] Все выходные данные / результаты описаны с примерами
- [ ] Взаимодействие с другими компонентами описано
- [ ] **Все failure modes описаны** — что происходит при каждом сбое на старте и в работе
- [ ] Для каждого события которое логируется — приведён пример лог-строки
- [ ] Конфиг-пример содержит только поля текущего MVP (не будущих фич)

## Нефункциональные требования

- [ ] Выбраны только релевантные категории НФТ (не добавлено ради галочки)
- [ ] Каждое НФТ конкретно и измеримо — не "система должна быть быстрой"

## Полнота

- [ ] Нет допущений которые сделаны молча — все неясности подняты пользователю
- [ ] Acceptance-writer не принесёт новых failure modes которых нет в spec
  _(мысленный прогон: "что acceptance может добавить нового?")_
- [ ] Spec не расширяет `PROJECT.md` тихо — если добавлено новое поле конфига, это отражено

## Согласование

- [ ] Черновик показан пользователю
- [ ] Подтверждение получено

## Движение дальше

- [ ] Черновик подтверждён пользователем
- [ ] Report написан — 01-spec/02-review может начинать
- [ ] Если входных данных недостаточно — записан `clarification`, возврат к 00-research/01-interview
