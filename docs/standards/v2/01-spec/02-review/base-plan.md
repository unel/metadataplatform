# Базовый план: 01-spec-review

## 1. Сбор контекста

- Прочитай spec из последнего `01-spec/01-write/report-*.md`
- Прочитай `PROJECT.md` — архитектура, стек, ограничения
- Прочитай `PLAN.md` — место фичи в roadmap
- Определи смежные фичи и прочитай их spec (если есть зависимости)

## 2. Анализ противоречий

- Внутренние: требования не противоречат друг другу внутри спеки
- Внешние: нет противоречий со spec смежных фич
- Архитектурные: нет противоречий с `PROJECT.md`

## 3. Проверка полноты ФТ

- Все входные данные описаны с примерами
- Поведение при невалидных входных данных описано
- Поведение при недоступности зависимостей описано
- Все failure modes описаны — старт и runtime
- Для каждого логируемого события — пример лог-строки
- Конфиг-пример содержит только поля текущего MVP
- Явно обозначено что не входит в MVP

## 4. Проверка полноты НФТ

- Все релевантные категории покрыты (не добавлено ради галочки)
- Каждое НФТ конкретно и измеримо

## 5. Проверка corner-cases

- Пустые входные данные
- Граничные значения
- Конкурентные запросы если применимо
- Частичный сбой в середине операции

## 6. Мысленный прогон

- Что acceptance-writer добавит нового, чего нет в spec? Если что-то — это замечание.
- Что тест-writer добавит нового, чего нет в spec? Если что-то — это замечание.

## 7. Формирование отчёта

- Каждое замечание: классификация (critical / warning / note) + объяснение + конкретная цитата из spec
- Если замечаний нет — явно написать "Спека чистая. Готово к acceptance."
- Статус: `done` если нет critical/warning, иначе `failed`

## Движение вперёд и назад

**Done когда:** нет замечаний уровня critical или warning
**Следующий:** 02-acceptance/01-write
**Передаём:** report-NNN.md

**Возврат назад:**
- есть critical/warning замечания → `failed`, переход к 01-spec/03-fix
- spec настолько неполна что ревью невозможно → `clarification`, возврат к 01-spec/01-write
