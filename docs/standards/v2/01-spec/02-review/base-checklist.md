# Базовый чек-лист: 01-spec-review

## Контекст

- [ ] Spec прочитана из последнего `01-spec/01-write/report-*.md`
- [ ] `PROJECT.md` прочитан
- [ ] `PLAN.md` прочитан
- [ ] Смежные spec прочитаны (если есть зависимости)

## Противоречия

- [ ] Требования не противоречат друг другу внутри спеки
- [ ] Нет противоречий со spec смежных фич
- [ ] Нет противоречий с архитектурой из `PROJECT.md`

## Полнота ФТ

- [ ] Все входные данные описаны с примерами
- [ ] Поведение при невалидных входных данных описано
- [ ] Поведение при недоступности каждой зависимости описано
- [ ] **Все failure modes описаны** — что происходит при каждом сбое на старте и в runtime
- [ ] Для каждого логируемого события — пример лог-строки приведён
- [ ] Конфиг-пример содержит только поля текущего MVP (не будущих фич)
- [ ] Явно обозначено что не входит в MVP

## Полнота НФТ

- [ ] Все релевантные категории НФТ покрыты (не добавлено ради галочки)
- [ ] Каждое НФТ конкретно и измеримо — не "система должна быть быстрой"

## Corner-cases

- [ ] Пустые / nil входные данные рассмотрены
- [ ] Граничные значения рассмотрены
- [ ] Конкурентные запросы рассмотрены (если применимо)
- [ ] Частичный сбой в середине операции рассмотрен

## Мысленный прогон

- [ ] Acceptance-writer не принесёт новых failure modes которых нет в spec
  _(что acceptance добавит нового? если что-то — это замечание к spec)_
- [ ] Тест-writer не принесёт новых edge cases которых нет в spec
  _(что тесты покроют сверх того что описано? если что-то важное — это замечание)_

## Качество отчёта

- [ ] Каждое замечание имеет классификацию: `critical` / `warning` / `note`
- [ ] Каждое замечание содержит объяснение и конкретную цитату или ссылку на пункт spec
- [ ] Если замечаний нет — явно написано "Спека чистая"
- [ ] Не переписана spec — только указаны проблемы

## Движение дальше

- [ ] Если нет critical/warning — report помечен `done`, следующий 02-acceptance/01-write
- [ ] Если есть critical/warning — report помечен `failed`, следующий 01-spec/03-fix
- [ ] Если spec неполна для ревью — записан `clarification`, возврат к 01-spec/01-write
