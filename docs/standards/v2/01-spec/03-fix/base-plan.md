# Базовый план: 01-spec-fix

## 1. Сбор контекста

- Прочитай актуальную spec: сначала ищи в `01-spec/03-fix/report-*.md` (последний), если нет — в `01-spec/01-write/report-*.md` (последний)
- Прочитай замечания: последний `01-spec/02-review/report-*.md`
- Прочитай `PROJECT.md` если замечания касаются архитектурных решений

## 2. Составление плана исправлений

Для каждого замечания (CR-N):
- Определи что именно нужно изменить
- Определи как это изменить (конкретная формулировка, добавление раздела, etc.)
- Если замечание требует архитектурного решения — отметь как "требует эскалации"

## 3. Согласование плана

- Покажи пользователю план: по каждому CR — что будет изменено
- Жди подтверждения или правок
- Если замечание требует архитектурного решения — остановись, эскалируй

## 4. Внесение исправлений

- Исправляй только то что указано в замечаниях — не переписывай spec целиком
- Каждое исправление — точечное: не трогай несмежные разделы

## 5. Формирование отчёта

Запиши отчёт содержащий:
- Полный обновлённый текст spec
- Changelog: по каждому CR — что изменено, или почему не изменено (с обоснованием)

## 6. Обновление статусов

После фикса:
- `01-spec/02-review` → pending (нужно перепроверить исправленную spec)
- Все последующие процессы (`02-acceptance`, `03-tests`, `04-code`, `05-docs`) → stale

## Движение вперёд и назад

**Done когда:** все critical/warning замечания закрыты, report написан
**Следующий:** 01-spec/02-review (повторный проход)
**Передаём:** report-NNN.md (содержит обновлённую spec + changelog)

**Возврат назад:**
- замечание требует архитектурного решения → `clarification`, эскалация к пользователю через оркестратора
