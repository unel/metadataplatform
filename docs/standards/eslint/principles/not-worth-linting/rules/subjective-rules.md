# Субъективные правила

**Категория:** правила которые отражают личный стиль а не объективное качество кода.

**Примеры категорий которые не стоит линтить:**
- Порядок ключей в объекте — группировка по смыслу часто важнее алфавита
- Предпочтение стрелочных функций везде — обычная функция иногда читаемее
- Требование деструктуризации везде — `const x = obj.x` иногда понятнее `const { x } = obj`
- Предпочтение `for...of` vs `forEach` — оба валидны

## Правило

Если правило вызывает споры в команде — скорее всего оно субъективное. Субъективные правила не линтить. Исключение: если команда явно договорилась и зафиксировала решение — тогда можно.
