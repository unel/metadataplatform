# Обработка неопределённости в скиллах

Когда скилл сталкивается с вопросом на который нет однозначного ответа — вызывай `/ask`.

## Не вызывай `/ask` если

- ответ не влияет на скоуп, архитектуру или другие компоненты
- есть очевидный дефолт (синяя или зелёная кнопка, json или yaml форматирование)
- ответ можно вывести из `CLAUDE.md`, спеки или уже принятых ADR

В этих случаях — прими решение сам, кратко объясни почему, продолжай.

## Вызывай `/ask` если

- вопрос влияет на дизайн, скоуп или несколько компонентов
- нет достаточного контекста чтобы принять решение самостоятельно
- пользователь ответил "не знаю / надо подумать" на прямой вопрос

## Что делать с вердиктом `/ask`

- **`decision`** — зафиксируй решение и продолжай (или передай наверх если ты не оркестратор)
- **`needs-rfc`** — передай причину наверх оркестратору, останови текущий flow

Записывать в `status.md` — ответственность вызывающего скилла или оркестратора, не `/ask`.
